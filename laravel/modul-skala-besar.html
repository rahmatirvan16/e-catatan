Ringkasan Minggu 10

Skala Besar

Semua yang dipelajari sejak Minggu 1 membangun satu aplikasi monolith (laravel-crud-app berisi produk, cart, order, payment dalam satu codebase). Minggu ini melihat bagaimana arsitektur berubah kalau skala sudah jauh lebih besar — dan kapan (atau kapan tidak) perubahan itu masuk akal.

Dari Monolith ke Microservices

AspekMonolithMicroservices
DeploymentSatu unit, deploy semua sekaligusTiap service deploy independen
ScalingScale seluruh aplikasiScale hanya service yang butuh (mis. hanya Payment saat flash sale)
DatabaseSatu database bersamaTiap service punya database sendiri
Kompleksitas operasionalRendahTinggi — butuh service discovery, distributed tracing, dll
Cocok untukTim kecil-menengah, kebanyakan startupTim besar, domain bisnis sudah sangat jelas batasannya
Microservices Bukan "Upgrade" Otomatis
Banyak tim memecah monolith terlalu dini dan malah menambah kompleksitas tanpa manfaat nyata. laravel-crud-app sebagai monolith modular (Repository/ Service Pattern dari Minggu 2) sudah cukup untuk sebagian besar skala bisnis. Microservices baru masuk akal kalau tim sudah besar dan bagian tertentu (misal Payment) butuh di-scale & di-deploy jauh lebih sering daripada bagian lain.

Memecah laravel-crud-app (Contoh Konseptual)

Kalau memang diperlukan, begini kira-kira laravel-crud-app dipecah berdasarkan bounded context (batas domain bisnis yang jelas):

laravel-crud-app (monolith)
  ├── Produk & Kategori
  ├── Cart
  ├── Order
  └── Payment

        ↓ dipecah berdasarkan bounded context ↓

produk-service      (Laravel API — kelola produk & stok)
order-service       (Laravel API — cart, checkout, status order)
payment-service     (Laravel API — integrasi Midtrans, webhook)
ServiceDatabase SendiriTanggung Jawab
produk-serviceMySQL produkCRUD produk, stok, kategori
order-serviceMySQL orderCart, checkout, state machine order
payment-serviceMySQL paymentIntegrasi Midtrans, verifikasi webhook
Repository/Service Pattern Mempermudah Migrasi Ini
Karena logic bisnis sudah rapi di ProdukService, OrderService, dan PaymentService sejak Minggu 2, "memisahkan" mereka jadi aplikasi Laravel terpisah jauh lebih mudah dibanding kalau semua logic tercampur langsung di controller.

Komunikasi Antar Service

PolaKapan DipakaiContoh
Synchronous (REST API)Butuh jawaban langsungorder-service panggil produk-service untuk cek stok saat checkout
Asynchronous (Message Broker)Tidak perlu jawaban langsung, tahan gangguan sementaraorder-service publish "OrderPaid", produk-service dengar & kurangi stok
Masalah Klasik: Distributed Transaction
Kalau checkout perlu "kurangi stok DI produk-service" DAN "buat order DI order-service" sebagai satu transaksi, DB::transaction() (yang dipakai di Minggu 7) tidak lagi cukup karena keduanya database berbeda. Di sinilah message broker dan event sourcing berperan.

Kenapa Butuh Message Broker?

Queue Redis (Minggu 3) sudah cukup untuk background job dalam satu aplikasi. Message broker (RabbitMQ, Kafka) dipakai saat pesan perlu mengalir antar service/aplikasi berbeda, dengan jaminan pengiriman yang lebih kuat.

AspekRedis Queue (Minggu 3)RabbitMQKafka
ModelAntrian sederhanaMessage queue dengan routing kompleksDistributed log/stream
Cocok untukBackground job dalam 1 appKomunikasi antar service, RPC-likeEvent stream volume tinggi, replay data
Retensi pesanHilang setelah diprosesHilang setelah di-ackBisa disimpan & diputar ulang (replay)

Pilih Salah Satu Dulu
Untuk kebanyakan project, cukup pilih RabbitMQ (lebih sederhana) atau Kafka (untuk volume event yang sangat tinggi dan butuh replay data) — jarang butuh keduanya sekaligus di satu project.

Publish & Consume Event (Contoh RabbitMQ)

app/Events/OrderPaid.php
class OrderPaid

    public function __construct(
        public readonly int $orderId,
        public readonly array $items   // [['produk_id' => 1, 'qty' => 2], ...]
    ) {}

Publish dari order-service
Queue::connection('rabbitmq')->pushRaw(
    json_encode([
        'event' => 'OrderPaid',
        'order_id' => $order->id,
        'items' => $order->items->toArray(),
    ]),
    'stock-updates'   // nama queue/exchange
);
Consume di produk-service
class ProcessStockUpdate implements ShouldQueue
{
    public function handle(array $payload): void
    {
        foreach ($payload['items'] as $item) {
            Produk::find($item['produk_id'])?->decrement('stok', $item['qty']);
        }
    }
}
Eventual Consistency
Stok baru berkurang beberapa saat setelah order dibayar, bukan seketika dalam satu transaction seperti Minggu 7. Ini disebut eventual consistency — trade-off yang harus diterima saat data tersebar di banyak service.

Konsep CQRS

CQRS (Command Query Responsibility Segregation): pisahkan jalur untuk menulis data (Command) dari jalur untuk membaca data (Query) — keduanya bisa dioptimasi dan di-scale secara independen. 

Model Biasa (yang dipakai sejak Minggu 1):
  Controller → Service → satu Model Eloquent → satu tabel (baca & tulis sama)

CQRS:
  Command (tulis) → OrderService → tabel "orders" (ternormalisasi, banyak relasi)
  Query (baca)    → OrderReadService → tabel "orders_view" (denormalisasi, siap tampil)
AspekTanpa CQRSDengan CQRS
Struktur data bacaSama dengan struktur tulis (perlu banyak JOIN)Bisa didenormalisasi khusus untuk kecepatan baca
ScalingBaca & tulis di-scale bersamaBaca (sering) bisa di-scale terpisah dari tulis (jarang)
KompleksitasRendahTinggi — perlu sinkronisasi data baca & tulis
Kapan CQRS Masuk Akal?
Hanya kalau pola baca dan tulis benar-benar sangat berbeda skalanya (mis. jutaan orang lihat halaman produk, tapi cuma ribuan yang checkout). Untuk sebagian besar fitur laravel-crud-app, Eloquent biasa (Minggu 1) sudah lebih dari cukup.

Event Sourcing Dasar

Alih-alih menyimpan status akhir (orders.status = 'paid'), Event Sourcing menyimpan riwayat semua kejadian yang membentuk status itu. Status akhir dihitung dengan memutar ulang (replay) semua event. 

Cara Biasa (Minggu 7):
  orders.status = 'paid'   ← hanya tahu status TERAKHIR

Event Sourcing:
  OrderCreated   { order_id: 1 }
  OrderPaid      { order_id: 1, paid_at: '...' }
  OrderShipped   { order_id: 1, shipped_at: '...' }
  → status "shipped" DIHITUNG dari riwayat 3 event di atas
KeuntunganKerugian
Riwayat lengkap untuk audit (siapa ubah apa, kapan)Query "status saat ini" jadi lebih kompleks (butuh replay atau proyeksi)
Bisa "time travel" ke state di masa laluStorage event terus bertambah, butuh strategi snapshot
Cocok untuk debugging masalah bisnis kompleksKurva belajar tim lebih curam

Implementasi Sederhana Event Sourcing di Laravel

database/migrations/..._create_order_events_table.php
Schema::create('order_events', function (Blueprint $table) {
    $table->id();
    $table->foreignId('order_id')->constrained();
    $table->string('event_type');   // OrderCreated, OrderPaid, OrderShipped
    $table->json('payload');
    $table->timestamp('occurred_at');
});
app/Services/OrderEventStore.php
class OrderEventStore
{
    public function record(int $orderId, string $eventType, array $payload): void
    {
        DB::table('order_events')->insert([
            'order_id'    => $orderId,
            'event_type'  => $eventType,
            'payload'     => json_encode($payload),
            'occurred_at' => now(),
        ]);
    }

    public function currentStatus(int $orderId): string
    {
        $lastEvent = DB::table('order_events')
            ->where('order_id', $orderId)
            ->orderByDesc('occurred_at')
            ->first();

        return match ($lastEvent->event_type ?? null) {
            'OrderShipped' => 'shipped',
            'OrderPaid'    => 'paid',
            default          => 'pending',
        };
    }
}
Mulai Kecil
Tidak perlu framework event sourcing penuh untuk mulai belajar konsepnya — tabel order_events sederhana seperti di atas sudah cukup untuk memahami prinsip "simpan riwayat, bukan cuma status akhir".

Kenapa Butuh Kubernetes?

docker compose (Minggu 8) cukup untuk menjalankan stack di satu mesin. Kubernetes (K8s) mengelola container di banyak mesin/server sekaligus — auto-scaling, auto-healing (restart otomatis container yang mati), dan rolling update tanpa downtime.

AspekDocker ComposeKubernetes
SkalaSatu mesinCluster banyak mesin
Auto-scalingTidak ada bawaanBawaan (Horizontal Pod Autoscaler)
Auto-healingTerbatas (restart: always)Otomatis reschedule ke node sehat
Cocok untukDevelopment, project kecil-menengahProduction skala besar, banyak service (microservices)
Tidak Selalu Perlu K8s
Kalau docker compose di satu server sudah menangani trafik dengan baik (divalidasi lewat k6 di Minggu 8), belum tentu perlu Kubernetes. K8s menambah kompleksitas operasional signifikan — masuk akal saat sudah benar-benar butuh multi-server & multi-service.

Deployment & Service YAML

Manifest Kubernetes mendefinisikan berapa banyak instance (Pod) aplikasi yang harus selalu berjalan, memakai image Docker yang sama dari Minggu 8.

k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: laravel-crud-app
spec:
  replicas: 3   # 3 instance identik berjalan bersamaan
  selector:
    matchLabels:
      app: laravel-crud-app
  template:
    metadata:
      labels:
        app: laravel-crud-app
    spec:
      containers:
        - name: app
          image: myusername/laravel-crud-app:latest
          ports:
            - containerPort: 9000
          envFrom:
            - secretRef:
                name: laravel-crud-app-secrets
k8s/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: laravel-crud-app-svc
spec:
  selector:
    app: laravel-crud-app
  ports:
    - port: 80
      targetPort: 9000
  type: LoadBalancer
Secrets Tetap Aman (Minggu 6)
secretRef mengambil environment variable dari Kubernetes Secret (dibuat terpisah lewat kubectl create secret), bukan ditulis langsung di file YAML yang bisa ter-commit ke Git.

Menjalankan di Minikube

Terminal
# Buat secret dari file .env (contoh sederhana)
kubectl create secret generic laravel-crud-app-secrets --from-env-file=.env

# Terapkan manifest
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

# Cek status Pod (harus 3 Running, sesuai "replicas: 3")
kubectl get pods

# Akses aplikasi dari Minikube
minikube service laravel-crud-app-svc

Simulasikan Auto-healing

Terminal
# Hapus satu Pod secara paksa
kubectl delete pod <nama-pod>

# Cek lagi — Kubernetes otomatis membuat Pod baru menggantikan yang dihapus
kubectl get pods
Ini Bedanya dengan Supervisor (Minggu 9)
Supervisor menjaga proses tetap hidup di satu server. Kubernetes menjaga jumlah Pod tetap sesuai replicas di seluruh cluster — kalau satu server mati total, Pod dipindah otomatis ke server lain yang masih hidup.
