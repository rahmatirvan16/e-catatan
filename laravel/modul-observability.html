Ringkasan Minggu 4

Observability

Kalau Minggu 1–3 fokus membangun fitur, Minggu 4 ini fokus supaya kamu bisa melihat apa yang sebenarnya terjadi di dalam aplikasi — request apa yang lambat, query apa yang berulang, dan error apa yang muncul di production, sebelum user yang mengeluh duluan.

Pengenalan Laravel Telescope

Telescope adalah dashboard debugging bawaan Laravel untuk environment local/development. Ia merekam semua request, query, job, exception, mail, dan cache yang terjadi, lalu menampilkannya di halaman web yang mudah dibaca.

Tanpa TelescopeDengan Telescope
Debug pakai dd() / dump() manualSemua request & query terekam otomatis
Cek query lambat lewat DB::listen() manualDurasi tiap query langsung terlihat di dashboard
Sulit lacak job queue yang gagalTab khusus untuk failed jobs + stack trace
Tidak tahu email apa saja yang terkirim saat testingPreview isi email yang dikirim, tanpa perlu inbox asli
Analogi
Kalau storage/logs/laravel.log itu seperti catatan tulisan tangan yang harus dibaca baris per baris, Telescope itu seperti dashboard mobil — semua indikator penting (request, query, error) tampil visual dan bisa difilter.

Install & Setup Telescope

Langkah 1 — Install via Composer

Terminal
composer require laravel/telescope --dev

Pakai flag --dev karena Telescope hanya dibutuhkan saat development, tidak perlu ikut ter-install di server production.

Langkah 2 — Install Assets & Migration

Terminal
php artisan telescope:install
php artisan migrate

Perintah ini membuat config/telescope.php dan tabel-tabel telescope_entries untuk menyimpan data yang direkam.

Langkah 3 — Hanya Aktif di Local

Supaya Telescope tidak jalan di production (dan tidak membebani server), daftarkan provider-nya secara kondisional.

app/Providers/AppServiceProvider.php
public function register(): void
{
    if ($this->app->environment('local')) {
        $this->app->register(\Laravel\Telescope\TelescopeServiceProvider::class);
        $this->app->register(\App\Providers\TelescopeServiceProvider::class);
    }
}

Langkah 4 — Akses Dashboard

Terminal
php artisan serve

Buka http://127.0.0.1:8000/telescope — dashboard langsung menampilkan aktivitas aplikasi begitu ada request masuk.

Watcher & Fitur Telescope

Telescope merekam data lewat "watcher" — tiap jenis data punya watcher-nya sendiri, bisa diaktif/nonaktifkan di config/telescope.php.

WatcherMerekam
RequestsSetiap HTTP request: method, URL, status, durasi, payload
QueriesSemua query SQL beserta durasi — berguna untuk cari N+1
JobsJob queue: status, waktu proses, exception jika gagal
ExceptionsSemua exception yang terlempar beserta stack trace lengkap
MailPreview email yang dikirim, tanpa perlu SMTP asli
CacheCache hit/miss — berguna verifikasi Cache::remember() bekerja
Scheduled TaskRiwayat command terjadwal (schedule:run)

Contoh Kasus — Cari Query Lambat

Buka tab Queries, urutkan berdasarkan durasi. Query yang butuh >100ms biasanya kandidat untuk ditambahkan index (lihat kembali Indexing Dasar di Minggu 1) atau di-cache dengan Redis (lihat Minggu 3).

Batasi Data yang Disimpan

config/telescope.php
'watchers' => [
    Watchers\QueryWatcher::class => [
        'enabled' => true,
        'slow'    => 50,   // tandai query > 50ms sebagai lambat
    ],
],
Jangan Biarkan Tabel Membengkak
Telescope menyimpan semua data di database. Jadwalkan pembersihan data lama secara rutin: php artisan telescope:prune --hours=48 (bisa ditaruh di routes/console.php sebagai scheduled task).

Proteksi Akses Dashboard (Production)

Kalau Telescope tetap ingin diaktifkan di staging/production (misal untuk debug masalah spesifik), wajib dibatasi hanya untuk user tertentu.

app/Providers/TelescopeServiceProvider.php
protected function gate(): void
{
    Gate::define('viewTelescope', function ($user) {
        return $user->hasRole('admin');   // pakai Spatie Roles dari Minggu 2
    });
}
Jangan Pernah Publik di Production
Dashboard Telescope menampilkan payload request, query SQL, bahkan isi email — termasuk data sensitif. Kalau tidak yakin butuh di production, cukup aktifkan di local saja seperti langkah sebelumnya.

Logging — Level & Channel

Laravel menggunakan Monolog di balik layar untuk semua penulisan log. Berbeda dengan Telescope (untuk development), log adalah catatan permanen yang wajib ada juga di production.

Level Log (dari Paling Ringan ke Paling Kritis)

LevelKapan Dipakai
debugDetail teknis untuk development, tidak perlu di production
infoKejadian normal yang informatif, misal "user login"
noticeKejadian normal tapi cukup penting untuk dicatat
warningSesuatu yang tidak ideal tapi belum error
errorError yang perlu diperhatikan tapi aplikasi masih jalan
criticalBagian penting aplikasi tidak berfungsi
alert / emergencyButuh tindakan segera — aplikasi/sistem down

Menulis Log

use Illuminate\Support\Facades\Log;

Log::info('Produk baru dibuat', ['produk_id' => $produk->id]);
Log::warning('Stok produk hampir habis', ['produk_id' => $produk->id, 'stok' => $produk->stok]);
Log::error('Gagal memproses pembayaran', ['order_id' => $order->id, 'exception' => $e->getMessage()]);

Konfigurasi Channel

config/logging.php
'default' => env('LOG_CHANNEL', 'stack'),

'channels' => [
    'stack' => [
        'driver'            => 'stack',
        'channels'          => ['single'],
        'ignore_exceptions' => false,
    ],
    'daily' => [
        'driver' => 'daily',
        'path'   => storage_path('logs/laravel.log'),
        'level'  => env('LOG_LEVEL', 'debug'),
        'days'   => 14,   // simpan 14 hari, hapus otomatis setelahnya
    ],
],
DriverKarakteristik
singleSemua log masuk ke satu file laravel.log yang terus membesar
dailyFile log baru setiap hari, otomatis hapus file lama (direkomendasikan)
slackKirim log (biasanya level critical ke atas) ke channel Slack
stackGabungkan beberapa channel sekaligus, misal daily + slack

Custom Log Channel

Kadang butuh log terpisah untuk domain tertentu, misal semua aktivitas pembayaran dicatat di file sendiri supaya mudah diaudit.

config/logging.php
'channels' => [
    // ... channel bawaan ...

    'payment' => [
        'driver' => 'daily',
        'path'   => storage_path('logs/payment.log'),
        'level'  => 'info',
        'days'   => 90,   // log pembayaran disimpan lebih lama
    ],
],
Penggunaan
Log::channel('payment')->info('Pembayaran berhasil', [
    'order_id' => $order->id,
    'amount'   => $order->total,
]);
Kapan Bikin Channel Terpisah?
Kalau sebuah domain (pembayaran, keamanan, integrasi API eksternal) butuh retensi log berbeda atau perlu diaudit terpisah dari log umum aplikasi.

Contextual Logging

Log tanpa konteks (cuma teks) sulit ditelusuri saat production ramai. Selalu sertakan data pendukung supaya satu baris log cukup untuk memahami kejadian, tanpa perlu menebak-nebak.

Kurang Baik
Log::error('Checkout gagal');
Lebih Baik — Ada Konteks
Log::error('Checkout gagal', [
    'user_id'    => $user->id,
    'order_id'   => $order->id,
    'total'      => $order->total,
    'payment_method' => $order->payment_method,
    'exception'  => $e->getMessage(),
]);

Context Global untuk Semua Log

Sejak Laravel 11, ada helper Context untuk menempelkan data ke semua log dalam satu request, tanpa perlu mengulang di setiap pemanggilan Log::.

app/Http/Middleware/AddRequestContext.php
use Illuminate\Support\Facades\Context;

public function handle($request, Closure $next)
{
    Context::add('request_id', (string) Str::uuid());
    Context::add('user_id', $request->user()?->id);

    return $next($request);
}

Setelah ini, setiap Log::info(...) di request yang sama otomatis menyertakan request_id dan user_id — sangat membantu melacak satu alur request penuh di log yang ramai.

Kenapa Butuh Sentry?

File log dan Telescope bagus untuk debugging manual, tapi keduanya pasif — kamu harus buka dan cek sendiri. Sentry adalah layanan error tracking yang aktif memberi tahu begitu ada exception baru di production, lengkap dengan stack trace, user yang terdampak, dan seberapa sering error itu terjadi.

Aspekstorage/logs/laravel.logSentry
Notifikasi real-timeTidak adaEmail/Slack begitu error terjadi
Grouping error serupaManual, baca satu-satuOtomatis dikelompokkan per jenis error
Riwayat frekuensiSulit dilihatGrafik kapan & seberapa sering terjadi
Konteks user & environmentManual ditambahkanOtomatis terekam (browser, OS, request)

Install & Konfigurasi Sentry

Langkah 1 — Buat Project di Sentry

Daftar di sentry.io, buat project baru dengan platform "Laravel", lalu salin DSN (alamat unik project kamu).

Langkah 2 — Install Package

Terminal
composer require sentry/sentry-laravel
php artisan sentry:publish --dsn=YOUR_DSN_HERE

Langkah 3 — Konfigurasi .env

.env
SENTRY_LARAVEL_DSN=https://xxxxxxxx@o000000.ingest.sentry.io/0000000
SENTRY_TRACES_SAMPLE_RATE=0.2

SENTRY_TRACES_SAMPLE_RATE mengatur berapa persen request yang direkam performanya (bukan cuma error) — 0.2 berarti 20% request untuk hemat kuota.

Langkah 4 — Daftarkan sebagai Log Channel

config/logging.php
'channels' => [
    'stack' => [
        'driver'   => 'stack',
        'channels' => ['daily', 'sentry'],
    ],
    'sentry' => [
        'driver'    => 'sentry',
        'level'     => 'error',   // hanya error ke atas yang dikirim ke Sentry
    ],
],
Kenapa Cuma Level error ke Atas?
Kalau semua level (termasuk debug/info) dikirim ke Sentry, kuota project cepat habis dan dashboard jadi ramai oleh noise yang tidak penting.

Custom Context & Breadcrumbs

Tambahkan informasi tambahan supaya laporan error di Sentry lebih mudah ditelusuri akar masalahnya.

app/Exceptions/Handler.php atau Controller
use Sentry\State\Scope;
use function Sentry\configureScope;

configureScope(function (Scope $scope) use ($request) {
    $scope->setUser([
        'id'    => $request->user()?->id,
        'email' => $request->user()?->email,
    ]);
    $scope->setTag('environment', app()->environment());
});

Breadcrumb mencatat jejak langkah sebelum error terjadi — berguna untuk melihat "apa yang dilakukan user sebelum crash".

ProdukService.php
use function Sentry\addBreadcrumb;
use Sentry\Breadcrumb;

addBreadcrumb(new Breadcrumb(
    Breadcrumb::LEVEL_INFO,
    Breadcrumb::TYPE_DEFAULT,
    'produk',
    'Produk mulai diproses',
    ['produk_id' => $data['id'] ?? null]
));

Testing Sentry

Pastikan integrasi bekerja sebelum mengandalkannya di production.

Terminal
php artisan sentry:test

Perintah ini mengirim exception dummy — cek dashboard Sentry, error percobaan harus muncul dalam beberapa detik.

Kirim Error Manual dari Kode

use Sentry\Laravel\Facade as Sentry;

try {
    $this->produkService->store($data);
} catch (\Throwable $e) {
    Sentry::captureException($e);
    throw $e;   // tetap lempar supaya error handler Laravel jalan normal
}
Alur Observability Lengkap
Telescope untuk debugging saat development, Log (Monolog) untuk catatan permanen di semua environment, Sentry untuk notifikasi aktif saat error terjadi di production. Ketiganya saling melengkapi, bukan saling menggantikan.
