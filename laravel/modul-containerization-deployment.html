Ringkasan Minggu 8

Containerization & Deployment

Sejauh ini laravel-crud-app jalan di XAMPP/WSL lokal. Minggu ini project yang sama dibungkus jadi container Docker supaya bisa jalan identik di laptop manapun dan di server production, lalu diuji otomatis dan di-deploy lewat CI/CD, dan diuji performanya dengan k6.

Kenapa Butuh Containerization?

Masalah klasik: "di laptop saya jalan, kok di server error?" — biasanya karena versi PHP, ekstensi, atau konfigurasi server berbeda antara environment development dan production.

Tanpa DockerDengan Docker
Install PHP, MySQL, Redis manual di tiap mesinSemua dependency terbungkus dalam image, tinggal docker run
Versi PHP developer A ≠ versi di serverVersi persis sama di semua environment
Setup ulang project baru = install ulang semua toolsClone repo + docker compose up saja
Sulit menjalankan banyak versi PHP untuk project berbedaTiap project punya container terisolasi sendiri
Analogi
Container itu seperti kontainer pengiriman kapal — ukuran & formatnya standar, jadi bisa dipindah antar truk, kapal, atau kereta apapun tanpa perlu dibongkar-pasang ulang isinya.

Install Docker Desktop (Windows)

Docker Desktop adalah cara termudah menjalankan Docker di Windows — sudah termasuk Docker Engine, CLI, dan Docker Compose sekaligus.

Langkah 1 — Cek Prasyarat

  • Windows 10/11 64-bit
  • WSL2 aktif — kalau sudah mengikuti instalasi WSL di Minggu 1, syarat ini sudah terpenuhi
  • Virtualization diaktifkan di BIOS (biasanya sudah default aktif di laptop modern)

Langkah 2 — Download & Install

Download installer dari docker.com/products/docker-desktop, jalankan, pastikan opsi "Use WSL 2 instead of Hyper-V" tercentang saat instalasi.

Langkah 3 — Verifikasi Instalasi

Terminal (buka baru setelah restart)
docker --version
docker compose version
Contoh Output
Docker version 27.x.x, build xxxxxxx
Docker Compose version v2.x.x

Langkah 4 — Tes Cepat

Terminal
docker run hello-world

Kalau muncul pesan "Hello from Docker!", instalasi berhasil dan Docker siap dipakai.

Docker Desktop Harus Menyala
Docker Desktop perlu dijalankan (ada ikon paus di system tray) sebelum perintah docker apapun bisa dipakai di terminal.

Dockerfile untuk Laravel

Dockerfile
FROM php:8.3-fpm

WORKDIR /var/www

# Install ekstensi PHP yang dibutuhkan Laravel
RUN apt-get update && apt-get install -y \
    libzip-dev unzip git curl libpng-dev \
    && docker-php-ext-install pdo_mysql zip gd

# Install Composer di dalam image
COPY --from=composer:2 /usr/bin/composer /usr/bin/composer

COPY . .

RUN composer install --no-dev --optimize-autoloader

RUN chown -R www-data:www-data /var/www/storage /var/www/bootstrap/cache

EXPOSE 9000
CMD ["php-fpm"]
.dockerignore
.git
.env
node_modules
vendor
storage/logs/*
tests
Kenapa --no-dev?
Package development (Pest, PHPUnit, Telescope dari Minggu 3–4) tidak perlu ikut di image production — membuat image lebih kecil dan lebih cepat di-build.

docker-compose.yml — Satu Stack Lengkap

Gabungkan semua service yang sudah dipakai project sejak Minggu 1–7: app (PHP-FPM), web server (Nginx), database (MySQL), dan cache/queue (Redis).

docker-compose.yml
services:
  app:
    build: .
    volumes:
      - .:/var/www
    depends_on:
      - mysql
      - redis

  nginx:
    image: nginx:alpine
    ports:
      - "8000:80"
    volumes:
      - .:/var/www
      - ./docker/nginx.conf:/etc/nginx/conf.d/default.conf
    depends_on:
      - app

  mysql:
    image: mysql:8
    environment:
      MYSQL_DATABASE: laravel_crud_app
      MYSQL_ROOT_PASSWORD: secret
    volumes:
      - mysql_data:/var/lib/mysql
    ports:
      - "3306:3306"

  redis:
    image: redis:alpine
    ports:
      - "6379:6379"

  horizon:
    build: .
    command: php artisan horizon
    volumes:
      - .:/var/www
    depends_on:
      - redis
      - mysql

volumes:
  mysql_data:
Ini Persis Skill Minggu 3
Service redis dan horizon di atas adalah container terpisah untuk fitur cache & queue yang sudah dipelajari di Minggu 3 — sekarang dijalankan sebagai proses terisolasi, bukan langsung di mesin lokal.

Menjalankan Stack

Terminal
# Build image & jalankan semua service di background
docker compose up -d --build

# Jalankan migration di dalam container app
docker compose exec app php artisan migrate

# Lihat log semua service
docker compose logs -f

# Masuk ke shell container app (mirip SSH ke server)
docker compose exec app bash

# Hentikan semua service
docker compose down

Buka http://localhost:8000 — aplikasi berjalan persis seperti di XAMPP/WSL, tapi sekarang seluruh stack (app, web server, database, redis, queue worker) terbungkus rapi dan bisa dipindahkan ke server manapun yang punya Docker.

Volume untuk Data Persisten
mysql_data di volumes: memastikan data database tidak hilang saat container di-restart. Tanpa ini, setiap docker compose down akan menghapus semua data.

Kenapa Butuh CI/CD?

CI (Continuous Integration): setiap kali kode di-push, otomatis dijalankan test & pengecekan kualitas. CD (Continuous Deployment): kalau semua pengecekan lolos, kode otomatis di-deploy ke server.

Tanpa CI/CDDengan CI/CD
Developer lupa jalankan test sebelum pushTest (Pest dari Minggu 3) otomatis jalan di setiap push
Deploy manual, rawan salah langkahDeploy otomatis & konsisten setiap saat
Kerentanan dependency (Minggu 6) terlewatcomposer audit otomatis jadi bagian pipeline
GitHub Actions Tidak Perlu Diinstall
Berbeda dari Docker/k6, GitHub Actions berjalan di server milik GitHub (cloud), bukan di komputer kamu. Yang perlu disiapkan hanya file workflow (.yml) di dalam repository — sama repo GitHub yang sudah dipakai sejak awal e-catatan ini untuk hosting modul.

Workflow Testing Otomatis (CI)

.github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      mysql:
        image: mysql:8
        env:
          MYSQL_ROOT_PASSWORD: secret
          MYSQL_DATABASE: laravel_crud_app_test
        ports: ["3306:3306"]

    steps:
      - uses: actions/checkout@v4

      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.3'

      - name: Install Dependencies
        run: composer install --prefer-dist --no-progress

      - name: Copy .env
        run: cp .env.example .env && php artisan key:generate

      - name: Cek Kerentanan Dependency
        run: composer audit

      - name: Jalankan Migration
        run: php artisan migrate --force

      - name: Jalankan Test (Pest)
        run: ./vendor/bin/pest
Menerapkan Minggu 3 & 6
Workflow ini menjalankan Feature Test yang ditulis di Minggu 3 dan composer audit dari Minggu 6 secara otomatis di setiap push — tidak lagi mengandalkan developer ingat menjalankannya manual.

Workflow Build & Deploy (CD)

.github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    needs: []   # bisa diatur agar menunggu job "test" dari ci.yml lolos dulu

    steps:
      - uses: actions/checkout@v4

      - name: Login ke Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build & Push Image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: myusername/laravel-crud-app:latest

      - name: Deploy ke Server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /var/www/laravel-crud-app
            docker compose pull
            docker compose up -d
            docker compose exec -T app php artisan migrate --force

Menyimpan Secrets dengan Aman

Kredensial (DOCKERHUB_TOKEN, SERVER_SSH_KEY, dst) disimpan di GitHub Repository Secrets, bukan ditulis langsung di file .yml — menerapkan prinsip secrets management dari Minggu 6.

  1. Buka repo GitHub → Settings → Secrets and variables → Actions
  2. Klik New repository secret, isi nama (mis. DOCKERHUB_TOKEN) dan nilainya
  3. Secret otomatis tersedia sebagai ${{ secrets.NAMA_SECRET }} di workflow

Install k6 (Windows)

k6 adalah tools open-source untuk load testing — mensimulasikan banyak user memanggil API sekaligus, untuk melihat seberapa kuat aplikasi menahan beban sebelum benar-benar dipakai banyak orang.

Opsi 1 — winget (Bawaan Windows 10/11)

PowerShell
winget install k6 --source winget

Opsi 2 — Chocolatey

PowerShell (Run as Administrator)
choco install k6

Opsi 3 — Download Manual

Download installer .msi dari k6.io/docs/get-started/installation kalau winget/choco tidak tersedia.

Verifikasi Instalasi

Terminal
k6 version
Contoh Output
k6 v0.5x.x (go1.2x, windows/amd64)

Skrip Load Test

Simulasikan banyak user mengakses endpoint /api/produk dan melakukan checkout, mirip kondisi flash sale.

tests/load/produk-load-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
    stages: [
        { duration: '30s', target: 50 },   // naik bertahap ke 50 virtual user
        { duration: '1m',  target: 50 },   // tahan di 50 user selama 1 menit
        { duration: '20s', target: 0 },    // turun bertahap ke 0
    ],
    thresholds: {
        http_req_duration: ['p(95)<500'],   // 95% request harus < 500ms
        http_req_failed:   ['rate<0.01'],   // error rate harus < 1%
    },
};

export default function () {
    const res = http.get('http://localhost:8000/api/produk', {
        headers: { Authorization: `Bearer ${__ENV.TOKEN}` },
    });

    check(res, {
        'status 200': (r) => r.status === 200,
        'response cepat': (r) => r.timings.duration < 500,
    });

    sleep(1);
}

Jalankan Load Test

Terminal
k6 run -e TOKEN=your_sanctum_token tests/load/produk-load-test.js
Jangan Load Test Server Production Orang Lain
Load test hanya boleh dijalankan ke server milik sendiri atau yang sudah diizinkan eksplisit — menembak server pihak lain tanpa izin bisa dianggap serangan (mirip DDoS).

Membaca Hasil Load Test

Contoh Output k6
     ✓ status 200
     ✓ response cepat

     checks.........................: 100.00% ✓ 3000 ✗ 0
     http_req_duration..............: avg=120ms  min=45ms  med=110ms  p(95)=310ms
     http_req_failed................: 0.00%   ✓ 0    ✗ 3000
     http_reqs.......................: 3000    50/s
MetrikArtiTarget Sehat
http_req_duration (p95)95% request selesai di bawah waktu ini< 500ms untuk API biasa
http_req_failedPersentase request yang gagal (error/timeout)< 1%
http_reqsTotal request & throughput (request/detik)Sesuai target kapasitas

Kalau Hasilnya Buruk?

  • Cek query lambat lewat Telescope (Minggu 4)
  • Pastikan endpoint yang sering diakses sudah di-cache dengan Redis (Minggu 3)
  • Cek apakah N+1 query muncul kembali (Minggu 1)
  • Pertimbangkan menambah instance server (dibahas di Minggu 9–10)
Load Testing Bagian dari CI/CD
Jalankan k6 dengan threshold di atas sebagai job tambahan di GitHub Actions sebelum deploy ke production — kalau performa turun drastis dibanding sebelumnya, deployment otomatis dibatalkan.
