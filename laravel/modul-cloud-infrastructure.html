Cloud & Infrastructure

Docker (Minggu 8) membungkus aplikasi, tapi container itu perlu tempat berjalan yang stabil di production. Minggu ini: web server production-grade (Nginx), penjaga proses background (Supervisor), penyedia server sungguhan (AWS/GCP), dan cara mengelola semuanya sebagai kode (Terraform).

Kenapa Nginx untuk Production?

Sejak Minggu 1 kita pakai Apache (bawaan XAMPP) untuk development. Di production, Nginx lebih umum dipakai di depan Laravel karena arsitekturnya lebih ringan untuk menangani banyak koneksi bersamaan.

AspekApache (XAMPP)Nginx
Model prosesSatu proses/thread per koneksiEvent-driven, satu proses tangani ribuan koneksi
Konsumsi memoriNaik signifikan dengan banyak koneksiTetap rendah & stabil
Eksekusi PHPModul mod_php tertanamProxy ke PHP-FPM terpisah
Cocok untukDevelopment lokal, kemudahan konfigurasiProduction dengan trafik tinggi
Tidak Perlu Migrasi Buru-buru
Apache tetap bisa dipakai di production untuk trafik kecil-menengah. Nginx jadi relevan saat aplikasi sudah diuji dengan k6 (Minggu 8) dan terbukti butuh menangani banyak koneksi bersamaan secara efisien.

Install & Konfigurasi Nginx

Server production umumnya Linux (Ubuntu). Langkah ini dijalankan di server, bisa dites dulu di WSL (sudah di-setup sejak Minggu 1).

Langkah 1 — Install Nginx & PHP-FPM

Terminal (Ubuntu/WSL)
sudo apt update
sudo apt install nginx php8.3-fpm -y

sudo systemctl status nginx    # pastikan "active (running)"

Langkah 2 — Server Block untuk Laravel

/etc/nginx/sites-available/laravel-crud-app
server {
    listen 80;
    server_name laravel-crud-app.com;
    root /var/www/laravel-crud-app/public;

    add_header X-Frame-Options "DENY";
    add_header X-Content-Type-Options "nosniff";

    index index.php;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/var/run/php/php8.3-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        include fastcgi_params;
    }

    location ~ /\.(?!well-known).* {
        deny all;   # blokir akses ke file dotfile seperti .env
    }
}

Langkah 3 — Aktifkan Konfigurasi

Terminal
sudo ln -s /etc/nginx/sites-available/laravel-crud-app /etc/nginx/sites-enabled/
sudo nginx -t             # cek syntax config sebelum reload
sudo systemctl reload nginx
Blok location ~ /\. Itu Penting
Tanpa baris ini, file seperti .env (berisi secrets dari Minggu 6) bisa diakses langsung lewat browser kalau tidak sengaja ter-expose.

Supervisor untuk Queue Worker & Horizon

php artisan horizon (Minggu 3) dan queue worker harus selalu berjalan di background. Kalau proses ini crash atau server restart, Supervisor yang menjaganya tetap hidup dan otomatis menyalakannya kembali.

Install Supervisor

Terminal
sudo apt install supervisor -y

Konfigurasi untuk Horizon

/etc/supervisor/conf.d/horizon.conf
[program:horizon]
process_name=%(program_name)s
command=php /var/www/laravel-crud-app/artisan horizon
autostart=true
autorestart=true
user=www-data
redirect_stderr=true
stdout_logfile=/var/www/laravel-crud-app/storage/logs/horizon.log
stopwaitsecs=3600

Jalankan & Verifikasi

Terminal
sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl start horizon

sudo supervisorctl status   # harus tampil "horizon RUNNING"
Auto-restart Kalau Crash
autorestart=true membuat Supervisor otomatis menyalakan ulang proses Horizon kalau tiba-tiba mati — tidak perlu ada yang login manual ke server tiap kali worker berhenti.

Pilihan Layanan Cloud

Dua provider paling umum untuk deploy aplikasi Laravel skala kecil-menengah:

KebutuhanAWSGCP
Server (compute)EC2Compute Engine
Database terkelolaRDS (MySQL/PostgreSQL)Cloud SQL
Object storage (upload file)S3Cloud Storage
Cache terkelolaElastiCache (Redis)Memorystore
Container orchestrationECS / EKSGKE
Tidak Perlu Kuasai Keduanya Sekaligus
Konsepnya mirip di semua provider (compute, managed database, object storage). Kuasai satu provider dengan baik dulu — skill-nya banyak yang transferable ke provider lain.

Install CLI (AWS/GCP)

CLI dipakai untuk mengelola resource cloud dari terminal, tanpa harus klik-klik di dashboard web setiap saat.

AWS CLI (Windows)

PowerShell
winget install Amazon.AWSCLI
Verifikasi & Konfigurasi
aws --version
aws configure   # masukkan Access Key ID, Secret Key, region (mis. ap-southeast-1)

Google Cloud CLI (Windows)

PowerShell
winget install Google.CloudSDK
Verifikasi & Login
gcloud --version
gcloud init          # login & pilih project GCP
gcloud auth login
Jangan Commit Access Key
Kredensial AWS/GCP hasil aws configure tersimpan di ~/.aws/credentials — folder ini di luar project Git, tapi tetap pastikan tidak pernah menyalin isinya ke dalam kode atau .env yang ter-commit (prinsip Minggu 6).

Provisioning Server & Database (Manual, Sebelum Terraform)

Sebelum masuk ke Infrastructure as Code, pahami dulu apa yang sebenarnya dibuat — supaya paham apa yang nanti "diotomatisasi" oleh Terraform.

Membuat EC2 Instance via AWS CLI

Terminal
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t3.small \
  --key-name laravel-crud-app-key \
  --security-groups laravel-crud-app-sg

Membuat Database RDS

Terminal
aws rds create-db-instance \
  --db-instance-identifier laravel-crud-app-db \
  --db-instance-class db.t3.micro \
  --engine mysql \
  --master-username admin \
  --master-user-password 'GantiPasswordIni!' \
  --allocated-storage 20
Ini Cara "Manual" — Mudah Salah & Sulit Diulang
Menjalankan perintah satu-satu seperti ini cukup untuk belajar konsep, tapi rawan human error dan tidak tercatat sebagai "dokumentasi hidup". Itulah masalah yang dipecahkan Terraform di bagian berikutnya.

Kenapa Butuh Infrastructure as Code?

Infrastructure as Code (IaC) menulis definisi infrastruktur (server, database, jaringan) sebagai file konfigurasi — bukan langkah manual di console/CLI yang mudah lupa atau beda antar environment.

Manual (Console/CLI)Terraform (IaC)
Langkah tidak terdokumentasi otomatisSemua definisi ada di file .tf, versi-terkontrol di Git
Sulit reproduksi environment identikterraform apply menghasilkan environment yang sama persis
Perubahan tidak ter-reviewterraform plan menampilkan preview perubahan sebelum dieksekusi
Rawan resource "yatim" yang lupa dihapusSemua resource tercatat, terraform destroy membersihkan rapi

Install Terraform (Windows)

Opsi 1 — winget

PowerShell
winget install HashiCorp.Terraform

Opsi 2 — Chocolatey

PowerShell (Run as Administrator)
choco install terraform

Verifikasi Instalasi

Terminal
terraform -version
Contoh Output
Terraform v1.x.x
on windows_amd64

Terraform untuk Laravel (AWS)

Definisikan server EC2 dan database RDS yang tadi dibuat manual, sekarang sebagai kode yang bisa diulang kapan saja.

main.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "ap-southeast-1"
}

resource "aws_instance" "laravel_app" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.small"
  key_name      = "laravel-crud-app-key"

  tags = {
    Name = "laravel-crud-app-server"
  }
}

resource "aws_db_instance" "laravel_db" {
  identifier        = "laravel-crud-app-db"
  engine            = "mysql"
  instance_class    = "db.t3.micro"
  allocated_storage = 20
  username          = "admin"
  password          = var.db_password   # dari variabel, bukan hardcoded
  skip_final_snapshot = true
}
variables.tf
variable "db_password" {
  description = "Password untuk RDS"
  type        = string
  sensitive   = true   // tidak tampil di log/output Terraform
}

Jalankan Terraform

Terminal
terraform init      # download provider AWS
terraform plan -var="db_password=GantiPasswordIni!"   # preview perubahan
terraform apply -var="db_password=GantiPasswordIni!"  # eksekusi (minta konfirmasi "yes")
Menerapkan Minggu 6 — Secrets di Variabel
db_password ditandai sensitive = true dan diisi lewat variabel (bukan ditulis langsung di main.tf) — sama prinsipnya dengan .env yang tidak boleh commit ke Git.

Membersihkan Resource

Terminal
terraform destroy -var="db_password=GantiPasswordIni!"
Hati-hati dengan terraform destroy
Perintah ini menghapus semua resource yang didefinisikan di file Terraform, termasuk database berisi data production. Selalu jalankan terraform plan dulu untuk memastikan tidak ada yang terhapus tidak sengaja.

Checklist Deploy Project E-commerce ke Cloud

  1. 1

    Provision Infrastruktur

    terraform apply — buat EC2/Compute Engine + RDS/Cloud SQL sesuai definisi di atas.

  2. 2

    Install Docker di Server

    SSH ke server, install Docker Engine (bukan Docker Desktop — itu khusus Windows/Mac).

  3. 3

    Deploy via CI/CD

    Push ke branch main — workflow deploy.yml dari Minggu 8 otomatis build image & SSH ke server baru ini.

  4. 4

    Setup Nginx + Supervisor

    Konfigurasi reverse proxy dan pastikan Horizon selalu jalan lewat Supervisor.

  5. 5

    Load Test Ulang

    Jalankan skrip k6 dari Minggu 8 mengarah ke domain production untuk validasi kapasitas server sungguhan.
