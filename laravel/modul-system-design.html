System Design

Modul terakhir roadmap ini — lebih konseptual dibanding minggu-minggu sebelumnya, tapi konsep-konsep ini yang sering ditanyakan di wawancara level senior/staff engineer. Tujuannya bukan menghafal, tapi memahami trade-off di balik setiap keputusan arsitektur skala besar.

Konsep Primary-Replica (Master-Slave) Replication

Satu database Primary menerima semua tulisan (write), lalu menyalin datanya ke satu atau lebih Replica yang hanya melayani bacaan (read). Ini memungkinkan skala baca jauh lebih tinggi tanpa membebani satu database saja. 

                 ┌──────────┐
   WRITE ────────▶│ Primary  │
                 └────┬─────┘
                      │ replikasi data
          ┌───────────┼───────────┐
          ▼           ▼           ▼
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │Replica 1│ │Replica 2│ │Replica 3│ ◀── READ (banyak query SELECT)
     └─────────┘ └─────────┘ └─────────┘
AspekTanpa ReplicationDengan Replication
Kapasitas bacaTerbatas satu serverBertambah linear dengan jumlah replica
Konsistensi dataSelalu terbaru (real-time)Ada replication lag beberapa milidetik-detik
KetahananSatu titik kegagalanReplica bisa dipromosikan jadi Primary jika Primary mati
Replication Lag Itu Nyata
Kalau user baru saja checkout (write ke Primary) lalu langsung refresh halaman order (read dari Replica), ada kemungkinan kecil order belum sempat ter-replikasi — user melihat data lama sesaat. Ini trade-off yang harus disadari, bukan bug.

Read/Write Splitting di Laravel

Laravel sudah mendukung konfigurasi Primary-Replica secara native — tidak perlu package tambahan.

config/database.php
'mysql' => [
    'read' => [
        'host' => [
            '192.168.1.10',   // Replica 1
            '192.168.1.11',   // Replica 2
        ],
    ],
    'write' => [
        'host' => ['192.168.1.1'],   // Primary
    ],
    'sticky'   => true,
    'driver'   => 'mysql',
    'database' => env('DB_DATABASE'),
    'username' => env('DB_USERNAME'),
    'password' => env('DB_PASSWORD'),
],

Setelah dikonfigurasi, semua kode yang sudah ada sejak Minggu 1 tidak perlu diubah sama sekali — Laravel otomatis mengarahkan query SELECT ke Replica dan INSERT/UPDATE/DELETE ke Primary.

Apa Itu 'sticky' => true?
Dalam satu request yang sama, kalau sudah ada operasi write, query read berikutnya di request itu juga diarahkan ke Primary — menghindari kasus "baru saja create data, langsung dibaca lagi tapi belum ter-replikasi".
Cara Membuktikan Tahap Ini Berhasil
Cek SHOW PROCESSLIST; (atau pg_stat_activity untuk Postgres) di masing-masing server saat aplikasi jalan — query SELECT harus muncul di replica, sementara INSERT/UPDATE hanya muncul di primary. Uji juga sticky: dalam satu request yang melakukan create() lalu langsung find() data yang baru dibuat, hasilnya harus ketemu (dibaca dari Primary) — bukan null karena kebetulan dibaca dari replica yang belum sempat menerima salinan data.

Konsep Database Sharding

Replication menyalin seluruh data ke banyak server (bagus untuk scale baca). Sharding berbeda: data dipecah (horizontal partitioning) menjadi bagian-bagian, tiap bagian disimpan di server terpisah — untuk scale baca dan tulis sekaligus. 

Replication:                      Sharding:
  Server A: semua data              Server A: user_id 1-1.000.000
  Server B: semua data (copy)       Server B: user_id 1.000.001-2.000.000
  Server C: semua data (copy)       Server C: user_id 2.000.001-3.000.000
Strategi Shard KeyContohRisiko
Range-basedID 1–1jt di shard A, 1jt+ di shard BShard baru selalu jadi "hot spot" untuk data baru
Hash-basedhash(user_id) % jumlah_shardData tersebar rata, tapi query rentang jadi sulit
Directory-basedTabel lookup terpisah: user_id → shard manaLookup table sendiri jadi titik kegagalan
Sharding Itu Keputusan Berat
JOIN antar shard tidak mungkin dilakukan di level database — harus digabung di level aplikasi. Sharding baru dipertimbangkan kalau satu server database sudah benar- benar mentok meski sudah dioptimasi (indexing, caching Redis, read replica).

Implementasi Sederhana Routing Shard

app/Services/ShardRouter.php
class ShardRouter
{
    private const TOTAL_SHARDS = 4;

    public function connectionFor(int $userId): string
    {
        $shardIndex = $userId % self::TOTAL_SHARDS;
        return "mysql_shard_{$shardIndex}";   // sesuai nama koneksi di config/database.php
    }
}
Penggunaan
$connection = $shardRouter->connectionFor($user->id);

$order = Order::on($connection)->find($orderId);
Konteksnya untuk laravel-crud-app
Sharding baru relevan kalau laravel-crud-app sudah punya jutaan order dan satu database MySQL (bahkan dengan replica) sudah tidak sanggup. Untuk skala sebagian besar bisnis, indexing (Minggu 1) + Redis cache (Minggu 3) + read replica di atas sudah lebih dari cukup.
Cara Membuktikan Tahap Ini Berhasil
Lewat tinker: (new \App\Services\ShardRouter())->connectionFor(5) dan connectionFor(9) harus mengembalikan nama koneksi yang sama (5 % 4 == 9 % 4 == 1), sementara connectionFor(6) harus beda. Simpan satu order lewat Order::on($connection)->create(...), lalu cek langsung di server database yang sesuai shard-nya — barisnya harus ada di situ saja, tidak di shard lain.

Algoritma Load Balancing

Sejak Kubernetes (Minggu 10) menjalankan banyak Pod laravel-crud-app sekaligus, dibutuhkan load balancer untuk menyebar trafik masuk secara merata ke semua instance.

AlgoritmaCara KerjaCocok Untuk
Round RobinBergiliran ke tiap server secara berurutanSemua server punya kapasitas setara
Least ConnectionsKirim ke server dengan koneksi aktif paling sedikitRequest dengan durasi bervariasi (ada yang lama, ada yang cepat)
IP HashClient yang sama selalu diarahkan ke server yang samaPerlu sticky session (jarang dibutuhkan kalau session di Redis)

Nginx sebagai Load Balancer

Nginx (dipasang di Minggu 9) tidak hanya jadi web server — bisa juga berperan sebagai load balancer di depan beberapa instance aplikasi.

/etc/nginx/nginx.conf
upstream laravel_backend {
    least_conn;   # algoritma least connections

    server 10.0.0.1:9000;
    server 10.0.0.2:9000;
    server 10.0.0.3:9000;
}

server {
    listen 80;

    location / {
        proxy_pass http://laravel_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

Health Check

Load balancer harus tahu instance mana yang sehat, supaya tidak mengirim trafik ke server yang sedang bermasalah.

# Endpoint health check bawaan Laravel (bootstrap/app.php: health: '/up')
curl http://10.0.0.1:9000/up
# 200 OK → server sehat, tetap masuk daftar upstream
# timeout/500 → load balancer berhenti kirim trafik ke server ini
Kubernetes Sudah Punya Load Balancer Bawaan
Kalau sudah pakai Kubernetes Service type: LoadBalancer (Minggu 10), load balancing antar Pod sudah otomatis ditangani. Konfigurasi Nginx manual di atas lebih relevan untuk setup tanpa Kubernetes (langsung di beberapa EC2/VM).
Cara Membuktikan Tahap Ini Berhasil
Tambahkan endpoint sementara yang mengembalikan hostname server (mis. return response()->json(['server' => gethostname()]);), lalu GET endpoint itu berkali-kali lewat load balancer — response harus bergantian antar 10.0.0.1/.2/.3, bukan selalu server yang sama (kecuali memang pakai IP Hash). Matikan satu server (docker stop atau hentikan PHP-FPM-nya), curl ke load balancer lagi — trafik harus otomatis berhenti diarahkan ke server yang mati itu begitu health check-nya gagal.

Distributed Systems & CAP Theorem

Setiap sistem terdistribusi (data tersebar di banyak server, seperti sejak Replication & Sharding di atas) hanya bisa menjamin dua dari tiga properti berikut secara bersamaan — ini yang disebut CAP Theorem.

HurufPropertiArti
CConsistencySemua node selalu punya data yang sama persis, kapan saja dibaca
AAvailabilitySetiap request selalu dapat respons (tidak error), meski ada node yang down
PPartition ToleranceSistem tetap jalan meski koneksi antar node terputus (network partition)
Partition Tolerance Sebenarnya Wajib
Di sistem terdistribusi nyata, jaringan pasti pernah bermasalah — jadi "P" bukan pilihan, melainkan kenyataan yang harus dihadapi. Pilihan sesungguhnya adalah antara C dan A saat partition terjadi.

Trade-off CAP di Dunia Nyata

PilihanSaat Network PartitionContoh Sistem
CP (Consistency + Partition Tolerance)Sebagian node menolak request demi data tetap konsistenMySQL/PostgreSQL dengan replikasi sinkron, sebagian besar RDBMS
AP (Availability + Partition Tolerance)Semua node tetap merespons, meski datanya mungkin belum sinkronDynamoDB, Cassandra, DNS

Kaitan dengan Modul-modul Sebelumnya

  • Redis cache (Minggu 3) — memilih Availability: data cache boleh sedikit stale demi respons cepat
  • Message broker (Minggu 10) — eventual consistency adalah konsekuensi memilih Availability di atas Consistency yang ketat
  • Database Primary-Replica — replication lag adalah harga yang dibayar untuk Availability/skala baca lebih tinggi
  • Payment/Order (Minggu 7) — sengaja pakai DB::transaction() untuk menjamin Consistency ketat, karena data uang tidak boleh eventual consistency
Tidak Ada Jawaban "Benar" Universal
CAP Theorem bukan soal mencari sistem yang "sempurna" — soal sadar trade-off apa yang diambil untuk kebutuhan bisnis tertentu. Data pembayaran butuh Consistency ketat; data rekomendasi produk bisa toleran terhadap sedikit inkonsistensi demi Availability yang lebih tinggi.
