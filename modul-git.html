Git — Version Control

Panduan lengkap belajar Git dari nol sampai level kerja profesional sehari-hari: konsep dasar, alur perpindahan file, branching, kolaborasi lewat remote, sampai teknik dan kebiasaan yang dipakai tim engineering berpengalaman.

Apa itu Git & Kenapa Penting?

Git adalah distributed version control system (DVCS) yang merekam setiap perubahan pada file project sebagai riwayat (history) yang bisa dilihat, dibandingkan, dan dikembalikan kapan saja. Setiap developer punya salinan penuh riwayat project di komputernya sendiri — tidak wajib selalu online seperti sistem versi lama.

Tanpa GitDengan Git
Backup manual: project-final-fix2.zipRiwayat perubahan tercatat rapi dengan pesan commit
Sulit kerja bareng tanpa saling menimpa fileBanyak orang bisa kerja paralel lewat branch
Susah tahu siapa mengubah baris kode managit blame menunjukkan penulis tiap baris
Sulit kembali ke versi sebelumnyagit checkout / git revert ke commit manapun
Analogi Sederhana
Git seperti fitur "track changes" pada Word, tapi jauh lebih kuat: bisa mencabang jadi beberapa versi paralel (branch), menggabungkan lagi (merge), dan menyimpan seluruh riwayat itu di banyak komputer sekaligus (distributed).

Instalasi & Konfigurasi Awal

Langkah 1 — Install Git

Di Windows, unduh installer dari situs resmi Git lalu jalankan, atau pakai Git Bash yang biasanya sudah terpasang bersama XAMPP/tools developer lain. Cek instalasi lewat terminal:

Terminal
git --version

Langkah 2 — Konfigurasi Identitas

Setiap commit menyimpan nama & email pembuatnya — wajib diisi sekali saja per komputer.

Terminal
git config --global user.name "Nama Kamu"
git config --global user.email "kamu@email.com"

Langkah 3 — Cek Konfigurasi

Terminal
git config --list
Tips
Gunakan email yang sama dengan akun GitHub/GitLab kamu supaya commit otomatis terhubung ke profil kamu di sana.

Konsep Dasar & Alur Perpindahan File

Inti dari Git ada di tiga area di komputer kamu, ditambah satu area di server (remote). Memahami perpindahan file antar area ini adalah kunci supaya tidak bingung dengan istilah add, commit, push, dan pull.

AreaFungsi
Working DirectoryFolder project asli tempat kamu mengedit file sehari-hari
Staging Area (Index)"Ruang tunggu" berisi perubahan yang sudah ditandai siap untuk di-commit
Local RepositoryDatabase riwayat commit tersimpan di folder .git di komputer kamu
Remote RepositorySalinan repository di server (GitHub/GitLab/Bitbucket) untuk berbagi dengan tim

Diagram Alur Perpindahan File

flowchart LR WD["Working Directory
(file yang diedit)"] STG["Staging Area
(index)"] REPO["Local Repository
(.git)"] REMOTE[("Remote Repository
(GitHub / GitLab)")] WD -- "git add" --> STG STG -- "git commit" --> REPO REPO -- "git push" --> REMOTE REMOTE -- "git pull / fetch + merge" --> REPO REPO -- "git checkout / restore" --> WD STG -- "git restore --staged" --> WD

Baca diagram di atas seperti ini: file yang kamu edit di Working Directory ditandai siap-simpan dengan git add ke Staging Area, lalu git commit menyimpannya permanen ke Local Repository sebagai satu titik riwayat baru. Setelah itu git push mengirim riwayat itu ke Remote Repository supaya tim lain bisa mengambilnya dengan git pull.

Kesalahan Umum Pemula
Lupa git add sebelum git commit membuat perubahan tidak ikut tersimpan, dan lupa git push membuat perubahan hanya ada di komputer sendiri — tidak terlihat oleh tim.

Perintah Dasar Git

Memulai Repository

Terminal — project baru
git init
Terminal — clone project yang sudah ada
git clone https://github.com/username/nama-repo.git

Melihat Status & Riwayat

Terminal
git status
git log
git log --oneline --graph --all
PerintahKegunaan
git statusLihat file mana yang berubah, belum di-add, atau sudah staged
git logLihat riwayat commit lengkap dengan hash, penulis, dan pesan
git diffLihat detail baris yang berubah, sebelum di-add

Menyimpan Perubahan

Terminal
git add nama-file.php
git add .
git commit -m "Tambah fitur login"
Pesan Commit yang Baik
Tulis dalam kalimat perintah singkat & jelas, misal "Tambah validasi email di form registrasi" — bukan sekadar "update" atau "fix".

Branching — Bekerja Paralel Tanpa Saling Tabrak

Branch adalah "cabang" riwayat yang terpisah dari branch utama (biasanya main), sehingga kamu bisa mengembangkan fitur baru tanpa mengganggu kode yang sudah stabil.

Terminal
git branch feature/login
git switch feature/login
git switch -c feature/login   # buat sekaligus pindah branch
git branch                    # lihat daftar branch
switch vs checkout
git switch adalah perintah modern khusus berpindah branch (lebih jelas), sedangkan git checkout lebih lama dan serba-bisa (branch, file, maupun commit). Keduanya masih umum dipakai di tim manapun.

Merging & Menyelesaikan Conflict

Setelah fitur di branch selesai, gabungkan kembali ke branch utama dengan merge.

Terminal
git switch main
git merge feature/login

Diagram Branch & Merge

gitGraph commit id: "init project" commit id: "setup struktur" branch feature/login checkout feature/login commit id: "buat form login" commit id: "tambah validasi" checkout main commit id: "perbaiki README" merge feature/login id: "merge fitur login" branch hotfix/bug-123 checkout hotfix/bug-123 commit id: "perbaiki bug kritis" checkout main merge hotfix/bug-123 id: "merge hotfix"

Menyelesaikan Merge Conflict

Conflict terjadi kalau dua branch mengubah baris yang sama. Git menandai bagian yang bentrok langsung di dalam file:

nama-file.php
<<<<<<< HEAD
kode versi branch main
=======
kode versi branch feature/login
>>>>>>> feature/login

Edit manual bagian tersebut sampai jadi satu versi yang benar, hapus penanda <<<<<<</=======/>>>>>>>, lalu selesaikan dengan:

Terminal
git add nama-file.php
git commit -m "Resolve conflict merge feature/login"
Jangan Panik Saat Conflict
Conflict adalah hal normal saat kolaborasi, bukan tanda kesalahan. Baca konteks kedua versi kode dengan tenang sebelum memutuskan versi mana yang dipakai.

Remote & Kolaborasi Tim

Remote adalah alamat repository di server (GitHub, GitLab, dsb.) yang jadi titik pertemuan seluruh tim.

Terminal
git remote add origin https://github.com/username/nama-repo.git
git push -u origin main
git pull origin main
git fetch origin
PerintahKegunaan
git pushKirim commit lokal ke remote
git fetchAmbil riwayat terbaru dari remote tanpa menggabungkannya ke branch lokal
git pullSama seperti fetch lalu langsung merge ke branch aktif

Diagram Push, Pull, Fetch

sequenceDiagram participant WD as Working Directory participant Staging as Staging Area participant Local as Local Repository participant Remote as Remote (GitHub) WD->>Staging: git add file.php Staging->>Local: git commit -m "pesan" Local->>Remote: git push origin main Note over Remote: Rekan tim push commit lain Remote-->>Local: git pull origin main Local-->>WD: hasil merge diterapkan
Kebiasaan Tim Profesional
Biasakan git pull sebelum mulai kerja dan sebelum push, supaya selalu bekerja di atas kode terbaru dan meminimalkan conflict besar.

Strategi Branching Profesional

Tim profesional biasanya menyepakati aturan penamaan & alur branch supaya rilis tetap rapi. Tiga strategi paling umum:

StrategiKarakteristik
Git FlowBranch main, develop, feature/*, release/*, hotfix/* — cocok untuk rilis terjadwal
GitHub FlowHanya main + feature/*, deploy langsung setelah merge — cocok untuk continuous deployment
Trunk-basedSemua developer commit langsung ke main dengan branch berumur sangat pendek, dibantu feature flag

Diagram Git Flow

flowchart TD main(("main")) develop(("develop")) feature["feature/xyz"] release["release/1.2.0"] hotfix["hotfix/1.2.1"] develop -- "branch" --> feature feature -- "merge" --> develop develop -- "branch saat siap rilis" --> release release -- "merge + tag" --> main release -- "merge balik" --> develop main -- "branch saat bug production" --> hotfix hotfix -- "merge" --> main hotfix -- "merge balik" --> develop
Pilih Sesuai Kebutuhan Tim
Tim kecil dengan deploy sering biasanya cocok dengan GitHub Flow; tim besar dengan jadwal rilis tetap sering pakai Git Flow.

Teknik Tingkat Lanjut — Rebase & Cherry-pick

Rebase — Merapikan Riwayat

git rebase memindahkan commit branch kamu supaya seolah dibuat di atas commit terbaru main, menghasilkan riwayat yang lurus tanpa banyak titik merge.

Terminal
git switch feature/login
git rebase main
Golden Rule Rebase
Jangan pernah rebase branch yang sudah di-push dan dipakai bersama orang lain — itu mengubah hash commit dan bisa merusak riwayat rekan tim.

Cherry-pick — Ambil Satu Commit Spesifik

Mengambil satu commit tertentu dari branch lain tanpa ikut merge seluruh branch.

Terminal
git cherry-pick a1b2c3d

Stash, Reset vs Revert, Tag, Reflog

Stash — Simpan Sementara

Menyimpan perubahan yang belum selesai tanpa commit, supaya bisa pindah branch dulu.

Terminal
git stash
git stash pop

Reset vs Revert

PerintahEfek
git reset --hard HEAD~1Menghapus commit terakhir dari riwayat (mengubah sejarah) — hindari di branch bersama
git revert HEADMembuat commit baru yang membatalkan efek commit sebelumnya (riwayat tetap utuh) — aman untuk branch bersama
Aturan Tim Profesional
Di branch yang sudah di-push dan dipakai bersama, selalu gunakan revert, bukan reset --hard.

Tag — Menandai Versi Rilis

Terminal
git tag v1.0.0
git push origin v1.0.0

Reflog — Menyelamatkan Commit yang "Hilang"

Reflog mencatat setiap perpindahan HEAD, termasuk commit yang tampak hilang setelah reset --hard yang salah.

Terminal
git reflog
git reset --hard HEAD@{2}

Best Practice Profesional

Konvensi Pesan Commit — Conventional Commits

Terminal
feat: tambah fitur login dengan Google OAuth
fix: perbaiki validasi email kosong
docs: update panduan instalasi
refactor: pindahkan logic ke ProdukService
test: tambah unit test untuk CartService
PrefixKapan Dipakai
featFitur baru
fixPerbaikan bug
docsPerubahan dokumentasi saja
refactorRapikan kode tanpa mengubah perilaku
testMenambah/memperbaiki test

.gitignore — Jangan Commit File yang Tidak Perlu

.gitignore
/vendor
/node_modules
.env
/storage/*.key
/public/hot
Jangan Pernah Commit Secret
File .env berisi kredensial database/API key — wajib masuk .gitignore sejak commit pertama project.

Pull Request & Code Review

Alih-alih langsung push ke main, tim profesional membuka Pull Request (PR) dari branch fitur, meminta review rekan tim, baru merge setelah disetujui — menjaga kualitas kode dan berbagi pengetahuan antar tim.

Troubleshooting Umum

MasalahSolusi
Salah pesan commit terakhirgit commit --amend -m "Pesan baru"
Ingin batalkan git addgit restore --staged nama-file.php
Ingin buang perubahan di working directorygit restore nama-file.php
Merge salah & ingin batalgit merge --abort
Rebase bermasalah & ingin batalgit rebase --abort

Ringkasan — Roadmap Pemula ke Profesional

  • 1

    Level Pemula

    Paham alur Working Directory → Staging → Local Repo, kuasai init, clone, add, commit, status, log.

  • 2

    Level Menengah

    Nyaman dengan branch, merge, resolve conflict, serta kolaborasi lewat remote, push, pull.

  • 3

    Level Lanjutan

    Mahir rebase, cherry-pick, stash, paham kapan pakai reset vs revert, serta menerapkan salah satu strategi branching (Git Flow / GitHub Flow / Trunk-based).

  • 4

    Level Profesional

    Konsisten memakai konvensi commit message, .gitignore yang rapi, selalu lewat Pull Request & code review, serta percaya diri memakai reflog untuk menyelamatkan kesalahan.
