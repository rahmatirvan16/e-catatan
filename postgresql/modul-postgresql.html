PostgreSQL — Database Dasar sampai Lanjutan

Panduan lengkap belajar PostgreSQL dari nol sampai level yang dipakai di production: instalasi, SQL dasar, integrasi penuh dengan Laravel, fitur khas Postgres (JSONB, full-text search), sampai indexing, performance tuning, backup, dan replikasi.

Apa itu PostgreSQL & Kenapa Penting?

PostgreSQL (sering disebut "Postgres") adalah object-relational database management system open-source yang dikenal karena kepatuhannya pada standar SQL, dukungan tipe data yang kaya, dan kemampuan menangani beban kerja yang kompleks — mulai dari aplikasi CRUD sederhana sampai sistem analitik skala besar.

Banyak tim memilih Postgres ketika kebutuhan aplikasi mulai melampaui apa yang mudah dilakukan MySQL: query JSON yang fleksibel, full-text search bawaan, index khusus (GIN/GiST/BRIN), dan jaminan konsistensi transaksi yang sangat ketat.

AspekMySQLPostgreSQL
Tipe data JSONJSON (tanpa index native yang efisien)JSONB biner, bisa di-index dengan GIN
Full-text searchTerbatas, butuh MATCH AGAINSTBawaan lewat tsvector/tsquery
ConcurrencyRow-level locking (tergantung engine)MVCC penuh sejak awal, minim blocking baca/tulis
EkstensibilitasTerbatasExtension (PostGIS, pgcrypto, pg_trgm, dll)
Constraint & tipe customDasarCHECK kompleks, domain, tipe data custom, array
Bukan "lebih baik" secara mutlak
MySQL tetap pilihan valid untuk kasus sederhana dengan tooling hosting yang lebih umum. Postgres unggul saat aplikasi butuh query data kompleks, integritas data ketat, atau fitur lanjutan seperti pencarian teks dan data semi-terstruktur (JSON).

Instalasi & Konfigurasi Awal

XAMPP secara default hanya menyertakan MySQL/MariaDB, jadi PostgreSQL perlu diinstal terpisah.

Langkah 1 — Install PostgreSQL

Di Windows, unduh installer resmi dari situs PostgreSQL (menyertakan pgAdmin sebagai GUI client) lalu jalankan wizard-nya. Saat instalasi, catat password superuser postgres dan port yang dipakai (default 5432).

Langkah 2 — Cek Instalasi via Terminal

Terminal
psql --version
psql -U postgres -h 127.0.0.1

Langkah 3 — Buat Role & Database Baru

Hindari memakai user postgres langsung untuk aplikasi. Buat role khusus dengan hak akses terbatas ke database aplikasinya saja.

psql
CREATE USER app_user WITH PASSWORD 'app_password';
CREATE DATABASE app_db OWNER app_user;
GRANT ALL PRIVILEGES ON DATABASE app_db TO app_user;
Tips
Gunakan pgAdmin atau ekstensi database client di editor kamu untuk eksplorasi visual, tapi tetap biasakan diri dengan perintah psql karena itu yang tersedia di server production lewat SSH.

CRUD Dasar (SELECT, INSERT, UPDATE, DELETE)

Sintaks dasar PostgreSQL sangat dekat dengan standar SQL ANSI.

Membuat Tabel

SQL
CREATE TABLE produk (
  id BIGSERIAL PRIMARY KEY,
  nama VARCHAR(150) NOT NULL,
  harga NUMERIC(12,2) NOT NULL DEFAULT 0,
  stok INTEGER NOT NULL DEFAULT 0,
  dibuat_pada TIMESTAMPTZ NOT NULL DEFAULT now()
);

INSERT

SQL
INSERT INTO produk (nama, harga, stok)
VALUES ('Keyboard Mekanik', 450000, 25)
RETURNING id;

RETURNING adalah fitur khas Postgres — langsung mengambil kolom hasil insert tanpa query SELECT tambahan.

SELECT

SQL
SELECT id, nama, harga
FROM produk
WHERE stok > 0
ORDER BY harga DESC
LIMIT 10;

UPDATE & DELETE

SQL
UPDATE produk SET stok = stok - 1 WHERE id = 1;
DELETE FROM produk WHERE stok = 0 AND dibuat_pada < now() - INTERVAL '1 year';
Cara Membuktikan Tahap Ini Berhasil
Jangan lanjut ke tipe data/relasi sebelum yakin sintaks dasarnya benar: \d produk di psql harus menampilkan struktur tabel sesuai CREATE TABLE di atas. Setelah INSERT, jalankan SELECT COUNT(*) FROM produk; — harus bertambah satu. Setelah UPDATE, SELECT stok FROM produk WHERE id = 1; harus menunjukkan angka yang sudah berkurang.

Tipe Data Khusus PostgreSQL

Selain tipe standar (INTEGER, VARCHAR, BOOLEAN), Postgres punya beberapa tipe data yang jarang ada di database lain.

TipeKegunaan
SERIAL / BIGSERIALAuto-increment integer, umum untuk primary key
NUMERIC(p,s)Angka presisi tetap, wajib untuk uang — hindari FLOAT
TEXTString tanpa batas panjang, tanpa penalti performa dibanding VARCHAR(n)
TIMESTAMPTZTimestamp yang menyimpan & menormalkan timezone — default pilihan untuk kolom waktu
UUIDIdentifier unik global, cocok untuk sistem terdistribusi
JSONBData semi-terstruktur dalam format biner, bisa di-index & di-query
ARRAYKolom berisi array dari tipe data lain, misal TEXT[]
ENUMTipe data custom dengan nilai terbatas, misal status pesanan
Hati-hati
Jangan pakai FLOAT/DOUBLE PRECISION untuk nilai uang — keduanya floating point dan rawan kesalahan pembulatan. Selalu pakai NUMERIC.

Primary/Foreign Key & Constraint

Constraint menjaga integritas data langsung di level database, sebagai lapisan pertahanan terakhir setelah validasi aplikasi.

SQL
CREATE TABLE kategori (
  id BIGSERIAL PRIMARY KEY,
  nama VARCHAR(100) NOT NULL UNIQUE
);

CREATE TABLE produk (
  id BIGSERIAL PRIMARY KEY,
  kategori_id BIGINT NOT NULL REFERENCES kategori(id) ON DELETE RESTRICT,
  nama VARCHAR(150) NOT NULL,
  harga NUMERIC(12,2) NOT NULL CHECK (harga >= 0),
  stok INTEGER NOT NULL DEFAULT 0 CHECK (stok >= 0)
);
ConstraintFungsi
PRIMARY KEYIdentitas unik & wajib per baris
FOREIGN KEY / REFERENCESMenjamin nilai kolom ada di tabel relasi
UNIQUEMencegah duplikasi nilai kolom
CHECKValidasi kondisi custom pada nilai kolom
NOT NULLKolom wajib diisi
ON DELETE CASCADE/RESTRICT/SET NULLAturan saat baris induk dihapus
Cara Membuktikan Tahap Ini Berhasil
Constraint baru benar-benar berguna kalau terbukti menolak data yang salah. Coba sengaja langgar masing-masing: insert kategori_id yang tidak ada di tabel kategori → harus gagal dengan error violates foreign key constraint. Insert harga negatif → harus gagal dengan violates check constraint. Kalau semua percobaan "salah" ini malah berhasil tersimpan, constraint-nya belum terpasang dengan benar.

Index Dasar (B-tree)

Index mempercepat pencarian dengan mengorbankan sedikit ruang penyimpanan dan kecepatan tulis. PRIMARY KEY dan UNIQUE otomatis membuat index B-tree.

SQL
CREATE INDEX idx_produk_kategori ON produk (kategori_id);
CREATE INDEX idx_produk_nama ON produk (nama);
Kapan Butuh Index?
Tambahkan index pada kolom yang sering dipakai di WHERE, JOIN, dan ORDER BY — terutama foreign key, karena Postgres tidak otomatis meng-index kolom foreign key seperti beberapa database lain.

BEGIN, COMMIT, ROLLBACK & ACID

Transaksi membungkus beberapa perintah SQL jadi satu unit kerja yang all-or-nothing — semuanya berhasil disimpan, atau semuanya dibatalkan kalau ada yang gagal. Ini fondasi dari properti ACID (Atomicity, Consistency, Isolation, Durability) yang jadi alasan utama orang tetap percaya pada database relasional untuk data penting seperti transaksi keuangan.

SQL
BEGIN;

UPDATE akun SET saldo = saldo - 100000 WHERE id = 1;
UPDATE akun SET saldo = saldo + 100000 WHERE id = 2;

-- Kalau salah satu UPDATE gagal (misal saldo jadi negatif karena CHECK constraint),
-- seluruh transaksi otomatis batal.
COMMIT;

Kalau terjadi kesalahan sebelum COMMIT, batalkan manual dengan ROLLBACK. SAVEPOINT berguna untuk membatalkan sebagian saja tanpa membatalkan seluruh transaksi.

SQL
BEGIN;
UPDATE akun SET saldo = saldo - 100000 WHERE id = 1;

SAVEPOINT sebelum_bonus;
UPDATE akun SET saldo = saldo + 5000 WHERE id = 1; -- bonus, ternyata salah aturan

ROLLBACK TO SAVEPOINT sebelum_bonus; -- batalkan bonus saja, transfer di atas tetap jalan
COMMIT;
Kesalahan Umum Pemula
Menganggap tiap statement Laravel (Model::create(), DB::table()->update()) otomatis aman digabung. Kalau ada beberapa perintah yang harus berhasil/gagal bersama-sama (misal kurangi stok + catat transaksi), bungkus dengan DB::transaction() di Laravel, yang di baliknya memakai BEGIN/COMMIT/ROLLBACK Postgres.
PHP
DB::transaction(function () {
    DB::table('produk')->where('id', 1)->decrement('stok');
    DB::table('transaksi')->insert(['produk_id' => 1, 'jumlah' => 1]);
});
Cara Membuktikan Tahap Ini Berhasil
Buktikan ROLLBACK benar-benar membatalkan: BEGIN, lakukan UPDATE, cek datanya sudah berubah lewat SELECT di sesi/tab yang sama, lalu ROLLBACK — SELECT ulang harus menunjukkan data kembali seperti semula. Ulangi dengan COMMIT di akhir — kali ini perubahan harus permanen bahkan setelah membuka sesi psql baru.

Isolation Level & Deadlock

Isolation level menentukan seberapa "terlihat" perubahan dari transaksi lain yang berjalan bersamaan. Postgres memakai MVCC (Multi-Version Concurrency Control) sehingga pembaca tidak pernah diblokir oleh penulis, tapi kamu tetap perlu memahami level isolasinya untuk kasus tertentu.

Isolation LevelPerilaku
READ COMMITTED (default)Tiap statement melihat data yang sudah ter-commit saat statement itu dijalankan — bisa berubah antar statement dalam transaksi yang sama
REPEATABLE READSeluruh transaksi melihat snapshot data yang konsisten sejak transaksi dimulai
SERIALIZABLELevel paling ketat — hasil akhir dijamin sama seperti transaksi dijalankan satu per satu (serial), tapi lebih rawan perlu retry saat konflik
SQL
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT stok FROM produk WHERE id = 1; -- misal 5
-- ...proses lain...
UPDATE produk SET stok = stok - 1 WHERE id = 1;
COMMIT;

Row Locking & Deadlock

SELECT ... FOR UPDATE mengunci baris agar transaksi lain menunggu sebelum bisa mengubahnya — berguna untuk mencegah race condition seperti dua request mengurangi stok yang sama secara bersamaan.

SQL
BEGIN;
SELECT stok FROM produk WHERE id = 1 FOR UPDATE; -- transaksi lain menunggu di sini
UPDATE produk SET stok = stok - 1 WHERE id = 1;
COMMIT;
Deadlock
Terjadi saat dua transaksi saling menunggu kunci yang dipegang satu sama lain (transaksi A menunggu baris yang dikunci B, sementara B menunggu baris yang dikunci A). Postgres otomatis mendeteksi dan membatalkan salah satu transaksi dengan error deadlock detected — aplikasi harus siap menangkap error ini dan retry. Hindari dengan selalu mengunci baris dalam urutan yang konsisten (misal selalu urut berdasarkan ID terkecil dulu).
Cara Membuktikan Tahap Ini Berhasil
Butuh dua sesi psql terbuka bersamaan untuk melihat locking sungguhan: di sesi A, BEGIN; SELECT * FROM produk WHERE id = 1 FOR UPDATE; (jangan COMMIT dulu). Di sesi B, coba UPDATE produk SET stok = stok - 1 WHERE id = 1; — sesi B harus menggantung (menunggu), bukan langsung sukses. Setelah sesi A COMMIT, sesi B baru lanjut. Kalau sesi B langsung sukses tanpa menunggu, berarti FOR UPDATE belum efektif.

Konfigurasi Koneksi Laravel → PostgreSQL

Pastikan ekstensi PHP pdo_pgsql dan pgsql aktif (di XAMPP, buka php.ini dan hapus tanda ; di depan kedua baris tersebut, lalu restart Apache).

.env

.env
DB_CONNECTION=pgsql
DB_HOST=127.0.0.1
DB_PORT=5432
DB_DATABASE=app_db
DB_USERNAME=app_user
DB_PASSWORD=app_password

config/database.php

Laravel sudah menyediakan koneksi pgsql secara default — cukup pastikan blok berikut ada dan cocok dengan .env:

config/database.php
'pgsql' => [
    'driver' => 'pgsql',
    'host' => env('DB_HOST', '127.0.0.1'),
    'port' => env('DB_PORT', '5432'),
    'database' => env('DB_DATABASE', 'forge'),
    'username' => env('DB_USERNAME', 'forge'),
    'password' => env('DB_PASSWORD', ''),
    'charset' => 'utf8',
    'search_path' => 'public',
    'sslmode' => 'prefer',
],
Tips
Uji koneksi cepat dengan php artisan tinker lalu jalankan DB::connection()->getPdo(); — kalau tidak ada error, koneksi berhasil.

Migrasi & Schema Builder

Migration Laravel bekerja sama untuk semua driver, tapi kamu bisa memakai tipe kolom khusus Postgres lewat schema builder.

database/migrations/xxxx_create_produk_table.php
Schema::create('produk', function (Blueprint $table) {
    $table->id();
    $table->foreignId('kategori_id')->constrained()->restrictOnDelete();
    $table->string('nama');
    $table->decimal('harga', 12, 2);
    $table->unsignedInteger('stok')->default(0);
    $table->jsonb('atribut')->nullable();
    $table->uuid('public_id')->unique();
    $table->timestampsTz();
});

jsonb() dan timestampsTz() hanya masuk akal dipakai kalau koneksi default kamu memang PostgreSQL — keduanya memetakan langsung ke tipe JSONB dan TIMESTAMPTZ.

Cara Membuktikan Tahap Ini Berhasil
Setelah php artisan migrate, jalankan \d produk di psql — kolom atribut harus bertipe jsonb (bukan json biasa) dan created_at/updated_at harus bertipe timestamp with time zone. Kalau tipenya salah, kemungkinan method Schema Builder yang dipakai keliru (mis. json() alih-alih jsonb()).

Eloquent & Query Builder Khusus Postgres

Eloquent tetap dipakai seperti biasa, tapi query builder menyediakan method tambahan untuk fitur khas Postgres seperti JSON.

PHP
// Query kolom JSONB
$produk = Produk::whereJsonContains('atribut->warna', 'merah')->get();

// Full-text search sederhana
$hasil = Produk::whereRaw(
    "to_tsvector('indonesian', nama) @@ plainto_tsquery('indonesian', ?)",
    [$keyword]
)->get();

// Insert dengan RETURNING otomatis lewat Eloquent
$produk = Produk::create([
    'nama' => 'Mouse Wireless',
    'harga' => 250000,
    'atribut' => ['warna' => 'hitam', 'garansi_bulan' => 12],
]);
Portabilitas
Method seperti whereJsonContains ditulis Laravel agar tetap bekerja di MySQL & Postgres, tapi query mentah (whereRaw) dengan sintaks tsvector sudah spesifik Postgres dan tidak portable.

Join, Subquery & CTE

Join

SQL
SELECT p.nama, k.nama AS kategori
FROM produk p
JOIN kategori k ON k.id = p.kategori_id
WHERE p.stok > 0;

Common Table Expression (CTE)

WITH membuat query sementara yang lebih mudah dibaca dibanding subquery bersarang, dan bisa dipakai berulang dalam query utama.

SQL
WITH penjualan_bulan_ini AS (
  SELECT produk_id, SUM(jumlah) AS total_terjual
  FROM transaksi
  WHERE dibuat_pada >= date_trunc('month', now())
  GROUP BY produk_id
)
SELECT p.nama, COALESCE(pb.total_terjual, 0) AS total_terjual
FROM produk p
LEFT JOIN penjualan_bulan_ini pb ON pb.produk_id = p.id
ORDER BY total_terjual DESC;

Window Functions

Window function menghitung agregat tanpa menggabungkan baris seperti GROUP BY — setiap baris asli tetap tampil, dengan kolom tambahan hasil perhitungan window.

SQL
SELECT
  nama,
  kategori_id,
  harga,
  RANK() OVER (PARTITION BY kategori_id ORDER BY harga DESC) AS rank_termahal,
  AVG(harga) OVER (PARTITION BY kategori_id) AS rata_rata_kategori
FROM produk;
Kapan Dipakai
Cocok untuk kebutuhan seperti "produk termahal per kategori", "running total", atau "ranking user per bulan" — hal yang sulit dilakukan dengan GROUP BY biasa karena butuh detail per baris sekaligus agregat.

View & Materialized View

View menyimpan definisi query sebagai "tabel virtual" yang selalu dihitung ulang saat diakses — berguna untuk menyederhanakan query kompleks yang sering dipakai berulang.

SQL
CREATE VIEW ringkasan_penjualan AS
SELECT p.id AS produk_id, p.nama, SUM(t.jumlah) AS total_terjual
FROM produk p
JOIN transaksi t ON t.produk_id = p.id
GROUP BY p.id, p.nama;

SELECT * FROM ringkasan_penjualan WHERE total_terjual > 100;

Materialized view menyimpan hasil query secara fisik seperti tabel biasa — jauh lebih cepat dibaca, tapi datanya tidak otomatis update dan perlu di-refresh manual.

SQL
CREATE MATERIALIZED VIEW ringkasan_penjualan_harian AS
SELECT date_trunc('day', t.dibuat_pada) AS tanggal, SUM(t.jumlah) AS total
FROM transaksi t
GROUP BY 1;

-- Jalankan berkala (cron/scheduler), idealnya dengan CONCURRENTLY agar tidak mengunci baca
REFRESH MATERIALIZED VIEW CONCURRENTLY ringkasan_penjualan_harian;
Kapan Pakai yang Mana
Pakai view biasa untuk menyederhanakan query yang datanya harus selalu real-time. Pakai materialized view untuk laporan/dashboard berat yang boleh sedikit tertinggal (misal di-refresh tiap jam), demi menghindari query agregat berat dijalankan berulang kali.

Function (PL/pgSQL)

Function menyimpan logika di sisi database — berguna untuk komputasi yang harus konsisten dipakai banyak query/aplikasi, atau untuk memindahkan proses berat agar tidak bolak-balik data ke aplikasi.

SQL
CREATE OR REPLACE FUNCTION hitung_diskon(harga NUMERIC, persen NUMERIC)
RETURNS NUMERIC AS $$
BEGIN
  IF persen < 0 OR persen > 100 THEN
    RAISE EXCEPTION 'Persen diskon tidak valid: %', persen;
  END IF;
  RETURN harga - (harga * persen / 100);
END;
$$ LANGUAGE plpgsql;

SELECT hitung_diskon(100000, 15); -- 85000
Jangan Berlebihan
Logika bisnis yang sering berubah (aturan harga, promo) lebih baik tetap di kode aplikasi (Laravel) supaya mudah di-test dan di-deploy lewat proses CI/CD biasa. Simpan di database hanya untuk logika yang benar-benar stabil dan erat kaitannya dengan integritas data.
Cara Membuktikan Tahap Ini Berhasil
SELECT hitung_diskon(100000, 15); harus mengembalikan persis 85000. Uji juga jalur errornya: SELECT hitung_diskon(100000, 150); harus gagal dengan pesan "Persen diskon tidak valid: 150" — bukan diam-diam mengembalikan angka negatif.

Trigger

Trigger menjalankan function secara otomatis saat event tertentu terjadi (INSERT/UPDATE/DELETE) — sering dipakai untuk audit log atau menjaga kolom turunan tetap konsisten.

SQL
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_produk_updated_at
BEFORE UPDATE ON produk
FOR EACH ROW
EXECUTE FUNCTION set_updated_at();
Tips
Laravel sudah menangani updated_at otomatis lewat Eloquent, jadi contoh di atas lebih relevan untuk perubahan lewat psql langsung atau aplikasi lain yang mengakses database yang sama. Trigger paling berguna sebagai jaring pengaman di level database untuk aturan yang tidak boleh dilanggar siapa pun, termasuk query manual.
Cara Membuktikan Tahap Ini Berhasil
Lewat psql: catat dulu nilai updated_at sebuah baris, lalu UPDATE produk SET stok = stok WHERE id = 1; (tanpa menyentuh updated_at secara eksplisit) — SELECT updated_at FROM produk WHERE id = 1; harus menunjukkan waktu yang baru saja berubah, bukti trigger jalan otomatis meski update dilakukan lewat SQL mentah, bukan Eloquent.

JSONB & Operator JSON

JSONB menyimpan JSON dalam bentuk biner yang sudah di-parse, sehingga query terhadap isinya jauh lebih cepat dibanding tipe JSON teks biasa, dan bisa di-index.

SQL
-- Ambil nilai field
SELECT atribut->>'warna' AS warna FROM produk WHERE id = 1;

-- Cek apakah JSON mengandung key/value tertentu
SELECT * FROM produk WHERE atribut @> '{"warna": "merah"}';

-- Update satu field dalam JSONB tanpa menimpa seluruh isi
UPDATE produk
SET atribut = jsonb_set(atribut, '{garansi_bulan}', '24')
WHERE id = 1;

-- Index GIN untuk mempercepat query JSONB
CREATE INDEX idx_produk_atribut ON produk USING GIN (atribut);
OperatorFungsi
->Ambil field, hasil tetap JSONB
->>Ambil field, hasil sebagai teks
@>Cek apakah JSON kiri mengandung JSON kanan
?Cek apakah key tertentu ada

Full-text Search

Postgres punya mesin pencarian teks bawaan lewat tipe tsvector (dokumen yang sudah di-normalisasi) dan tsquery (kata kunci pencarian).

SQL
-- Pencarian langsung
SELECT nama FROM produk
WHERE to_tsvector('indonesian', nama) @@ plainto_tsquery('indonesian', 'keyboard mekanik');

-- Kolom tersimpan (generated column) + index GIN agar cepat di tabel besar
ALTER TABLE produk ADD COLUMN nama_tsv tsvector
  GENERATED ALWAYS AS (to_tsvector('indonesian', nama)) STORED;
CREATE INDEX idx_produk_nama_tsv ON produk USING GIN (nama_tsv);

SELECT nama FROM produk
WHERE nama_tsv @@ plainto_tsquery('indonesian', 'keyboard mekanik');
Untuk Kebutuhan Search yang Sangat Berat
Full-text search bawaan Postgres cukup untuk kebanyakan aplikasi. Kalau butuh fitur seperti typo-tolerance canggih atau skala jutaan dokumen dengan relevansi kompleks, pertimbangkan Elasticsearch/Meilisearch sebagai pelengkap.

Array & Composite Types

Kolom array berguna untuk data multi-nilai sederhana tanpa perlu tabel relasi terpisah — misalnya daftar tag.

SQL
ALTER TABLE produk ADD COLUMN tags TEXT[];

UPDATE produk SET tags = ARRAY['gaming', 'wireless', 'best-seller'] WHERE id = 1;

-- Cari produk yang punya tag tertentu
SELECT nama FROM produk WHERE 'gaming' = ANY(tags);

-- Pecah array jadi baris
SELECT nama, unnest(tags) AS tag FROM produk;
Kapan Pakai Array vs Tabel Relasi
Array cocok untuk data sederhana yang tidak butuh query relasional sendiri (tidak perlu di-join, tidak butuh atribut tambahan per tag). Kalau tag butuh metadata sendiri (warna, deskripsi, dsb), tetap gunakan tabel relasi biasa.

Role, GRANT & REVOKE

Jangan pernah memakai role superuser (postgres) untuk koneksi aplikasi. Buat role dengan hak akses seminimal mungkin sesuai kebutuhan — prinsip least privilege.

SQL
-- Role hanya boleh baca (untuk laporan/analytics)
CREATE ROLE readonly_app LOGIN PASSWORD 'secret';
GRANT CONNECT ON DATABASE app_db TO readonly_app;
GRANT USAGE ON SCHEMA public TO readonly_app;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_app;

-- Role aplikasi utama, tanpa hak DROP/ALTER tabel
CREATE ROLE app_writer LOGIN PASSWORD 'secret2';
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_writer;

-- Cabut akses tertentu
REVOKE DELETE ON transaksi FROM app_writer;
Kenapa Penting
Kalau kredensial aplikasi bocor, kerusakan yang bisa ditimbulkan terbatas sesuai hak akses role-nya. Role read-only untuk dashboard analytics, misalnya, tidak akan pernah bisa dipakai untuk mengubah data produksi meskipun kredensialnya bocor.

Row-Level Security (RLS)

Row-Level Security membatasi baris mana saja yang boleh dilihat/diubah oleh role tertentu — sangat berguna untuk aplikasi multi-tenant di mana satu tabel dipakai bersama oleh banyak pelanggan/organisasi.

SQL
ALTER TABLE pesanan ENABLE ROW LEVEL SECURITY;

CREATE POLICY pesanan_isolasi_tenant ON pesanan
  USING (tenant_id = current_setting('app.current_tenant')::bigint);

Aplikasi cukup mengatur variabel sesi sebelum query, dan Postgres otomatis menyaring baris di belakang layar tanpa perlu menambahkan WHERE tenant_id = ? di setiap query manual:

PHP
DB::statement("SET app.current_tenant = ?", [$tenantId]);
$pesanan = DB::table('pesanan')->get(); // otomatis hanya baris milik tenant ini
Bukan Pengganti Validasi Aplikasi
RLS adalah lapisan pertahanan tambahan di database, bukan pengganti pengecekan otorisasi di kode aplikasi. Tetap validasi akses di level aplikasi (misal policy Laravel), dan gunakan RLS sebagai jaring pengaman terakhir kalau ada bug atau akses langsung ke database yang terlewat dari lapisan aplikasi.
Cara Membuktikan Tahap Ini Berhasil
SET app.current_tenant = 1; lalu SELECT * FROM pesanan; harus hanya menampilkan baris milik tenant 1, meski tabelnya berisi data banyak tenant. Ganti SET app.current_tenant = 2; di sesi yang sama → hasil SELECT yang sama harus berubah total ke data tenant 2. Kalau baris tenant lain masih ikut muncul, policy RLS belum aktif atau kondisinya salah.

Membaca EXPLAIN ANALYZE

EXPLAIN ANALYZE menjalankan query sungguhan dan menampilkan rencana eksekusi beserta waktu nyata tiap langkah — alat utama untuk diagnosis query lambat.

SQL
EXPLAIN ANALYZE
SELECT * FROM produk WHERE kategori_id = 3 ORDER BY harga DESC;
IstilahArti
Seq ScanMembaca seluruh tabel baris demi baris — tanda index belum dipakai/tidak ada
Index ScanMencari lewat index lalu ambil baris dari tabel
Index Only ScanSemua data diambil langsung dari index, tanpa sentuh tabel — paling cepat
costEstimasi biaya relatif (bukan waktu nyata) sebelum eksekusi
actual timeWaktu nyata dalam milidetik saat query benar-benar dijalankan
Perhatian
Jangan jalankan EXPLAIN ANALYZE untuk query UPDATE/DELETE di production tanpa hati-hati — perintah ini benar-benar mengeksekusi query, bukan sekadar simulasi.
Cara Membuktikan Tahap Ini Berhasil
Jalankan EXPLAIN ANALYZE pada query yang sama sebelum dan sesudah membuat index di kolom kategori_id (lihat Index Dasar). Sebelum ada index, plan-nya harus menunjukkan Seq Scan; sesudahnya harus berubah jadi Index Scan, dan angka actual time harus lebih kecil. Kalau plan tidak berubah, kemungkinan tabelnya masih terlalu kecil untuk Postgres memilih pakai index (planner sengaja pilih Seq Scan kalau lebih murah untuk tabel kecil).

Index Lanjutan (GIN, GiST, BRIN, Partial, Composite)

Jenis IndexCocok Untuk
B-tree (default)Perbandingan =, <, >, ORDER BY
GINJSONB, array, full-text search — data dengan banyak nilai per baris
GiSTData geometris/spasial (PostGIS), pencarian rentang & kemiripan
BRINTabel sangat besar dengan data urut alami (misal kolom timestamp log)

Partial Index

Index hanya untuk subset baris yang relevan — lebih kecil & lebih cepat.

SQL
CREATE INDEX idx_produk_aktif ON produk (nama) WHERE stok > 0;

Composite Index

Index atas lebih dari satu kolom, urutan kolom penting — taruh kolom dengan filter kesetaraan (=) di depan.

SQL
CREATE INDEX idx_produk_kategori_harga ON produk (kategori_id, harga DESC);

Tips Optimasi Query

  • Hindari SELECT * — ambil hanya kolom yang dibutuhkan.
  • Index kolom foreign key secara eksplisit — Postgres tidak membuatnya otomatis.
  • Jalankan VACUUM & ANALYZE secara rutin (biasanya otomatis lewat autovacuum) agar statistik query planner tetap akurat dan ruang bekas baris terhapus bisa dipakai ulang.
  • Gunakan connection pooling (misal PgBouncer) saat aplikasi punya banyak koneksi pendek — membuka koneksi baru ke Postgres relatif mahal.
  • Pantau query lambat lewat ekstensi pg_stat_statements.
  • Batasi hasil besar dengan LIMIT + keyset pagination (bukan OFFSET besar) untuk data yang terus bertambah.

Partition by Range/List/Hash

Saat sebuah tabel tumbuh sampai puluhan/ratusan juta baris, query dan maintenance (vacuum, index rebuild) mulai melambat meski sudah punya index yang tepat. Table partitioning memecah satu tabel logis menjadi beberapa tabel fisik yang lebih kecil di belakang layar, tapi tetap terasa seperti satu tabel dari sisi query.

SQL
CREATE TABLE log_aktivitas (
  id BIGSERIAL,
  pesan TEXT,
  dibuat_pada TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (dibuat_pada);

CREATE TABLE log_aktivitas_2026_01 PARTITION OF log_aktivitas
  FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE TABLE log_aktivitas_2026_02 PARTITION OF log_aktivitas
  FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

Query yang menyertakan filter tanggal otomatis hanya membaca partisi yang relevan (partition pruning), bukan seluruh tabel:

SQL
EXPLAIN SELECT * FROM log_aktivitas
WHERE dibuat_pada >= '2026-02-01' AND dibuat_pada < '2026-02-15';
-- planner hanya menyentuh log_aktivitas_2026_02, bukan partisi Januari
JenisCocok Untuk
RANGEData berbasis rentang waktu/angka — log, transaksi per bulan
LISTData dengan kategori diskrit — misal partisi per region/negara
HASHDistribusi merata tanpa pola alami — sekadar memecah beban tulis
Kapan Baru Butuh Ini
Jangan mulai dari partitioning — ini kompleksitas tambahan yang hanya sepadan saat tabel benar-benar besar (biasanya puluhan juta baris ke atas) atau butuh menghapus data lama dengan cepat (DROP TABLE satu partisi jauh lebih cepat daripada DELETE jutaan baris).

pg_dump & pg_restore

Terminal
# Backup satu database ke file custom-format (terkompresi, bisa restore parsial)
pg_dump -U app_user -F c -f app_db.backup app_db

# Restore ke database baru
pg_restore -U postgres -d app_db_baru app_db.backup

# Backup semua database + role di server (untuk migrasi server)
pg_dumpall -U postgres -f semua_db.sql
Tips
Jadwalkan pg_dump otomatis (cron/Task Scheduler) dan simpan hasilnya di lokasi terpisah dari server database — backup yang hanya ada di server yang sama tidak melindungi dari kegagalan disk/server itu sendiri.
Cara Membuktikan Tahap Ini Berhasil
Backup yang belum pernah dicoba di-restore bukanlah backup yang bisa diandalkan. Setelah pg_restore ke app_db_baru, jalankan SELECT COUNT(*) FROM produk; di database baru itu dan bandingkan dengan jumlah baris di database asal — angkanya harus sama persis. Lakukan ini secara rutin, bukan cuma sekali saat setup awal.

Replikasi Dasar (Streaming Replication)

Streaming replication menyalin perubahan dari server primary ke satu atau lebih server replica secara nyaris real-time lewat write-ahead log (WAL).

graph LR A[Aplikasi] -->|Tulis| P[(Primary)] P -->|WAL Streaming| R1[(Replica 1)] P -->|WAL Streaming| R2[(Replica 2)] A -->|Baca| R1 A -->|Baca| R2

Replica biasanya dipakai untuk:

  • Scaling baca — arahkan query SELECT berat ke replica.
  • High availability — replica bisa dipromosikan jadi primary baru saat primary gagal.
  • Backup tanpa membebani primary — jalankan pg_dump dari replica.
Perlu Diingat
Replica bersifat read-only dan datanya sedikit tertinggal (replication lag). Jangan arahkan query yang butuh data paling baru (misal setelah user baru saja submit form) ke replica tanpa strategi tambahan.

Best Practice Umum

  • Gunakan snake_case untuk nama tabel & kolom (konvensi Postgres, juga default Laravel).
  • Selalu definisikan NOT NULL dan CHECK di level database, jangan andalkan validasi aplikasi saja.
  • Beri nama eksplisit pada constraint & index (mis. idx_produk_kategori) agar mudah dicari di migrasi berikutnya.
  • Pisahkan role database per aplikasi/tim dengan hak akses minimum yang dibutuhkan.
  • Aktifkan pg_stat_statements di lingkungan production untuk memantau query yang paling sering/lambat.
  • Uji migrasi skema besar di staging dengan volume data mirip production sebelum dijalankan di server nyata.

    Percaya diri mengelola backup & restore, memahami dasar replikasi untuk scaling & high availability, serta rutin memonitor performa lewat pg_stat_statements.
