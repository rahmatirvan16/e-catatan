Ringkasan Minggu 2

Arsitektur Kode & Otorisasi

Lanjutan dari project laravel-crud-app di Minggu 1. Kali ini fokus merapikan arsitektur kode dengan Repository/Service Pattern, menambah lapisan Middleware & Validation, mengatur hak akses dengan Spatie Roles & Permission, dan melindungi API dengan Rate Limiting.

Kenapa Butuh Repository Pattern?

Di Minggu 1, controller memanggil Eloquent langsung: Produk::all(), Produk::create(), dst. Untuk aplikasi kecil ini tidak masalah, tapi seiring project membesar, muncul beberapa keluhan umum:

  • Logic query tersebar di banyak controller — sulit dicari & diubah konsisten
  • Sulit di-unit test karena controller terikat langsung ke database
  • Kalau suatu saat sumber data berubah (misal pindah ke API eksternal atau cache), harus ubah banyak tempat

Repository Pattern menaruh semua query database di balik sebuah interface. Controller/Service hanya bicara dengan interface itu, tidak peduli implementasinya Eloquent, query builder, atau sumber lain.

AspekTanpa RepositoryDengan Repository
Query tersebarYa, di banyak controllerTerpusat di satu class repository
TestingSulit — butuh database asliMudah — repository bisa di-mock
Ganti sumber dataUbah semua controller terkaitCukup ubah satu implementasi
Kompleksitas awalRendahSedikit lebih tinggi (butuh interface + binding)
Alur Arsitektur Baru
Controller → Service (business logic) → Repository Interface → Eloquent Repository → Database

Interface & Repository

Langkah 1 — Buat Interface

app/Repositories/ProdukRepositoryInterface.php
<?php

namespace App\Repositories;

use App\Models\Produk;
use Illuminate\Database\Eloquent\Collection;

interface ProdukRepositoryInterface
{
    public function all(): Collection;
    public function find(int $id): ?Produk;
    public function create(array $data): Produk;
    public function update(Produk $produk, array $data): Produk;
    public function delete(Produk $produk): bool;
}

Langkah 2 — Implementasi Eloquent

Terminal
mkdir -p app/Repositories
app/Repositories/EloquentProdukRepository.php
<?php

namespace App\Repositories;

use App\Models\Produk;
use Illuminate\Database\Eloquent\Collection;

class EloquentProdukRepository implements ProdukRepositoryInterface
{
    public function all(): Collection
    {
        return Produk::with('kategori')->latest()->get();
    }

    public function find(int $id): ?Produk
    {
        return Produk::find($id);
    }

    public function create(array $data): Produk
    {
        return Produk::create($data);
    }

    public function update(Produk $produk, array $data): Produk
    {
        $produk->update($data);
        return $produk;
    }

    public function delete(Produk $produk): bool
    {
        return $produk->delete();
    }
}

Langkah 3 — Bind Interface ke Implementasi

Laravel perlu tahu implementasi mana yang dipakai setiap kali ada yang minta ProdukRepositoryInterface. Daftarkan di service container.

app/Providers/AppServiceProvider.php
use App\Repositories\ProdukRepositoryInterface;
use App\Repositories\EloquentProdukRepository;

public function register(): void
{
    $this->app->bind(
        ProdukRepositoryInterface::class,
        EloquentProdukRepository::class
    );
}
Keuntungan Binding Ini
Kalau nanti butuh ganti implementasi (misal CachedProdukRepository yang membungkus Redis cache), cukup ubah baris bind() ini — tidak perlu sentuh Controller atau Service sama sekali.

Service Layer

Service menyimpan business logic — aturan bisnis yang lebih dari sekadar simpan/ambil data. Repository hanya urusan data, Service yang mengatur "aturannya".

app/Services/ProdukService.php
<?php

namespace App\Services;

use App\Models\Produk;
use App\Repositories\ProdukRepositoryInterface;
use Illuminate\Validation\ValidationException;

class ProdukService
{
    public function __construct(
        private readonly ProdukRepositoryInterface $produkRepository
    ) {}

    public function getAll()
    {
        return $this->produkRepository->all();
    }

    public function store(array $data): Produk
    {
        // business rule: stok awal tidak boleh melebihi batas gudang
        if ($data['stok'] > 10000) {
            throw ValidationException::withMessages([
                'stok' => 'Stok awal melebihi kapasitas gudang.',
            ]);
        }

        return $this->produkRepository->create($data);
    }

    public function update(Produk $produk, array $data): Produk
    {
        return $this->produkRepository->update($produk, $data);
    }

    public function destroy(Produk $produk): bool
    {
        return $this->produkRepository->delete($produk);
    }
}
Kenapa Bukan Langsung di Repository?
Repository harusnya hanya urusan penyimpanan data (CRUD murni). Aturan bisnis seperti validasi kapasitas gudang, hitung diskon, atau kirim notifikasi setelah produk dibuat, tempatnya di Service — supaya repository tetap bisa dipakai ulang di konteks lain tanpa terikat satu aturan bisnis tertentu.

Refactor Controller Jadi Tipis

Controller sekarang hanya menerima request, memanggil service, dan mengembalikan response — tidak ada lagi logic bisnis atau query di dalamnya.

app/Http/Controllers/Api/ProdukApiController.php
<?php

namespace App\Http\Controllers\Api;

use App\Http\Controllers\Controller;
use App\Http\Requests\StoreProdukRequest;
use App\Http\Requests\UpdateProdukRequest;
use App\Models\Produk;
use App\Services\ProdukService;

class ProdukApiController extends Controller
{
    public function __construct(
        private readonly ProdukService $produkService
    ) {}

    public function index()
    {
        return response()->json([
            'success' => true,
            'data'    => $this->produkService->getAll(),
        ]);
    }

    public function store(StoreProdukRequest $request)
    {
        $produk = $this->produkService->store($request->validated());

        return response()->json(['success' => true, 'data' => $produk], 201);
    }

    public function update(UpdateProdukRequest $request, Produk $produk)
    {
        $produk = $this->produkService->update($produk, $request->validated());

        return response()->json(['success' => true, 'data' => $produk]);
    }

    public function destroy(Produk $produk)
    {
        $this->produkService->destroy($produk);

        return response()->json(['success' => true]);
    }
}
Laravel Otomatis Suntik Dependency
Karena ProdukRepositoryInterface sudah di-bind() ke EloquentProdukRepository, dan ProdukService butuh interface itu di constructor-nya, Laravel otomatis merangkai semuanya (Service → Repository → Controller) tanpa kode tambahan — ini disebut Dependency Injection.

Custom Middleware

Middleware menyaring request sebelum sampai ke controller. Laravel sudah punya middleware bawaan (auth:sanctum, throttle), tapi kita juga bisa buat middleware sendiri untuk aturan khusus aplikasi.

Contoh Kasus — Blokir User yang Di-nonaktifkan

Terminal
php artisan make:middleware EnsureUserIsActive
app/Http/Middleware/EnsureUserIsActive.php
<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;

class EnsureUserIsActive
{
    public function handle(Request $request, Closure $next)
    {
        if ($request->user()?->status === 'nonaktif') {
            return response()->json([
                'success' => false,
                'message' => 'Akun kamu dinonaktifkan, hubungi admin.',
            ], 403);
        }

        return $next($request);
    }
}

Daftarkan Alias Middleware

bootstrap/app.php
use App\Http\Middleware\EnsureUserIsActive;

->withMiddleware(function (Middleware $middleware) {
    $middleware->statefulApi();
    $middleware->alias([
        'active' => EnsureUserIsActive::class,
    ]);
})

Pakai di Routes

routes/api.php
Route::middleware(['auth:sanctum', 'active'])->group(function () {
    Route::apiResource('produk', ProdukApiController::class);
});

Form Request Validation

Di Minggu 1, validasi ditulis langsung di controller dengan $request->validate(). Cara ini cepat tapi bikin controller gemuk. Form Request memindahkan aturan validasi ke class terpisah.

Terminal
php artisan make:request StoreProdukRequest
php artisan make:request UpdateProdukRequest
app/Http/Requests/StoreProdukRequest.php
<?php

namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;

class StoreProdukRequest extends FormRequest
{
    public function authorize(): bool
    {
        return $this->user() !== null;
    }

    public function rules(): array
    {
        return [
            'nama_produk' => 'required|string|max:255',
            'harga'       => 'required|numeric|min:1',
            'stok'        => 'required|integer|min:0',
            'deskripsi'   => 'nullable|string|max:500',
            'kategori_id' => 'nullable|exists:kategoris,id',
        ];
    }

    public function messages(): array
    {
        return [
            'nama_produk.required' => 'Nama produk wajib diisi.',
            'harga.min'            => 'Harga minimal Rp1.',
        ];
    }
}

Controller tinggal type-hint class ini — validasi otomatis jalan sebelum method controller dieksekusi, dan gagal validasi otomatis mengembalikan 422 dengan detail error di response()->json().

Controller (cuplikan)
public function store(StoreProdukRequest $request)
{
    // $request->validated() sudah pasti data yang lolos aturan di atas
    $produk = $this->produkService->store($request->validated());
    return response()->json(['success' => true, 'data' => $produk], 201);
}
AspekValidasi di ControllerForm Request
ControllerGemuk, campur validasi & logicTipis, hanya panggil service
Reuse aturanCopy-paste antar methodSatu class, dipakai ulang
Custom pesan errorManual di setiap tempatTerpusat di messages()
Cek otorisasiManual ifMethod authorize() bawaan

Install Spatie Laravel-Permission

spatie/laravel-permission adalah package paling populer untuk mengatur roles (peran, misal admin/staff) dan permissions (izin granular, misal produk.delete) di Laravel.

Langkah 1 — Install via Composer

Terminal
composer require spatie/laravel-permission

Langkah 2 — Publish Migration & Config

Terminal
php artisan vendor:publish --provider="Spatie\Permission\PermissionServiceProvider"
php artisan migrate

Ini membuat tabel roles, permissions, model_has_roles, model_has_permissions, dan role_has_permissions.

Langkah 3 — Tambahkan Trait ke Model User

app/Models/User.php
use Spatie\Permission\Traits\HasRoles;   // tambahkan ini
use Laravel\Sanctum\HasApiTokens;

class User extends Authenticatable
{
    use HasApiTokens, HasRoles;   // tambahkan HasRoles

    protected $fillable = ['name', 'email', 'password'];
}

Setup Roles & Permissions

Buat Seeder

Terminal
php artisan make:seeder RolePermissionSeeder
database/seeders/RolePermissionSeeder.php
<?php

namespace Database\Seeders;

use Illuminate\Database\Seeder;
use Spatie\Permission\Models\Permission;
use Spatie\Permission\Models\Role;

class RolePermissionSeeder extends Seeder
{
    public function run(): void
    {
        // Buat permission granular
        $permissions = [
            'produk.view', 'produk.create', 'produk.update', 'produk.delete',
        ];
        foreach ($permissions as $permission) {
            Permission::firstOrCreate(['name' => $permission]);
        }

        // Buat role dan tempel permission-nya
        $admin = Role::firstOrCreate(['name' => 'admin']);
        $admin->givePermissionTo($permissions);   // admin dapat semua izin

        $staff = Role::firstOrCreate(['name' => 'staff']);
        $staff->givePermissionTo(['produk.view', 'produk.create', 'produk.update']);
        // staff tidak boleh hapus produk
    }
}
Terminal
php artisan db:seed --class=RolePermissionSeeder

Assign Role ke User

Tinker
php artisan tinker
>>> $user = User::find(1);
>>> $user->assignRole('admin');
>>> $user->getRoleNames();   // ["admin"]
>>> $user->getAllPermissions()->pluck('name');
KonsepContohFungsi
Roleadmin, staffKelompok besar hak akses (peran)
Permissionproduk.deleteIzin granular untuk satu aksi spesifik
Role → Permission$role->givePermissionTo(...)Satu role bisa punya banyak permission
User → Role$user->assignRole(...)Satu user bisa punya banyak role

Middleware Otorisasi di Routes

Daftarkan Middleware Spatie

bootstrap/app.php
use Spatie\Permission\Middleware\RoleMiddleware;
use Spatie\Permission\Middleware\PermissionMiddleware;

->withMiddleware(function (Middleware $middleware) {
    $middleware->statefulApi();
    $middleware->alias([
        'active'     => \App\Http\Middleware\EnsureUserIsActive::class,
        'role'       => RoleMiddleware::class,
        'permission' => PermissionMiddleware::class,
    ]);
})

Terapkan di Routes

routes/api.php
// Hanya admin yang boleh hapus produk
Route::middleware(['auth:sanctum', 'role:admin'])->group(function () {
    Route::delete('/produk/{produk}', [ProdukApiController::class, 'destroy']);
});

// Butuh permission spesifik, boleh dari role manapun yang punya izin itu
Route::middleware(['auth:sanctum', 'permission:produk.update'])->group(function () {
    Route::put('/produk/{produk}', [ProdukApiController::class, 'update']);
});
Response Kalau Ditolak
Kalau user tidak punya role/permission yang sesuai, Spatie melempar 403 Forbidden otomatis — tidak perlu cek manual di controller.

Cek Permission di Dalam Kode

Selain lewat middleware di routes, kamu juga bisa cek role/permission langsung di controller atau Blade untuk kasus yang lebih kondisional.

Di Controller

public function destroy(Produk $produk)
{
    if (! $request->user()->can('produk.delete')) {
        abort(403, 'Kamu tidak punya izin menghapus produk.');
    }

    $this->produkService->destroy($produk);
    return response()->json(['success' => true]);
}

Di Blade (kalau ada tampilan web)

@can('produk.delete')
  <button class="btn btn-danger">Hapus</button>
@endcan

@role('admin')
  <a href="/admin/laporan">Laporan Admin</a>
@endrole
MethodFungsi
$user->hasRole('admin')Cek apakah user punya role tertentu
$user->can('produk.delete')Cek permission (native Laravel Gate, didukung Spatie)
$user->hasPermissionTo('produk.delete')Sama seperti can(), versi Spatie
@can(...) / @endcanDirective Blade untuk sembunyikan elemen UI

Rate Limiting — Throttle Bawaan

Rate limiting membatasi berapa kali satu user/IP boleh memanggil endpoint dalam periode waktu tertentu — mencegah abuse dan brute-force.

Middleware throttle Bawaan Laravel

Grup route api di Laravel 12 secara default sudah dibungkus throttle:api (60 request/menit per user atau IP).

routes/api.php
// Custom limit langsung di route: 5 percobaan login per menit
Route::post('/login', [AuthController::class, 'login'])
    ->middleware('throttle:5,1');   // 5 request per 1 menit

Response Saat Limit Tercapai

HTTP 429 Too Many Requests
{
  "message": "Too Many Attempts."
}

Header response juga menyertakan X-RateLimit-Limit, X-RateLimit-Remaining, dan Retry-After supaya client tahu kapan boleh mencoba lagi.

Kenapa Endpoint Login Perlu Limit Ketat?
Tanpa rate limit, penyerang bisa mencoba ribuan kombinasi password per detik (brute-force). Limit 5 percobaan/menit membuat serangan seperti ini jadi sangat lambat dan tidak praktis.

Custom Rate Limiter

Untuk aturan yang lebih fleksibel (misal limit berbeda untuk user premium vs biasa), definisikan rate limiter sendiri lewat RateLimiter::for().

app/Providers/AppServiceProvider.php
use Illuminate\Cache\RateLimiting\Limit;
use Illuminate\Support\Facades\RateLimiter;
use Illuminate\Http\Request;

public function boot(): void
{
    RateLimiter::for('produk-api', function (Request $request) {
        $user = $request->user();

        // user dengan role "admin" dapat kuota lebih besar
        if ($user?->hasRole('admin')) {
            return Limit::perMinute(120)->by($user->id);
        }

        return Limit::perMinute(30)->by($user?->id ?: $request->ip());
    });
}
routes/api.php
Route::middleware(['auth:sanctum', 'throttle:produk-api'])->group(function () {
    Route::apiResource('produk', ProdukApiController::class);
});
MethodFungsi
Limit::perMinute($n)Batas per menit
Limit::perHour($n)Batas per jam
->by($key)Kunci pembatas — per user ID, per IP, atau kombinasi
Limit::none()Tanpa batas (misal untuk role tertentu)
Kombinasi dengan Otorisasi
Rate limiter custom di atas menunjukkan bagaimana Middleware, Spatie Roles, dan Rate Limiting bisa saling melengkapi — satu function bisa mengecek role user untuk menentukan kuota rate limit yang berbeda.
