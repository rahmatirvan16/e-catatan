Minggu 1 sudah membahas tiga ancaman dasar (CSRF, XSS, SQL Injection). Minggu ini melihat gambaran lebih luas lewat OWASP Top 10 — daftar risiko keamanan web paling kritis versi OWASP — lalu masuk ke enkripsi, manajemen secrets, dan pengerasan (hardening) API secara menyeluruh.

OWASP Top 10 — Pemetaan ke Laravel

OWASP Top 10 adalah daftar 10 risiko keamanan aplikasi web paling kritis, disusun berdasarkan data ribuan aplikasi nyata. Laravel sudah menyediakan proteksi bawaan untuk sebagian besar, tapi tetap perlu dikonfigurasi dan dipakai dengan benar.

A01: Broken Access Control

Risiko #1 di OWASP Top 10 — user bisa mengakses data/aksi yang seharusnya tidak boleh mereka akses.

Kasus Umum — IDOR (Insecure Direct Object Reference)

Rentan
public function show(int $id)
{
    // Siapapun yang login bisa lihat order siapapun cuma ganti angka di URL!
    $order = Order::find($id);
    return response()->json($order);
}
Aman — Policy Otorisasi
php artisan make:policy OrderPolicy --model=Order
// app/Policies/OrderPolicy.php
public function view(User $user, Order $order): bool
{
    return $user->id === $order->user_id || $user->hasRole('admin');
}
// Controller
public function show(Order $order)
{
    $this->authorize('view', $order);   // lempar 403 kalau tidak lolos
    return response()->json($order);
}
PrinsipPenerapan
Deny by defaultRoute dilindungi middleware auth, hanya buka yang benar-benar publik
Cek kepemilikan resourcePolicy view/update/delete, bukan cuma cek login
Jangan percaya input clientID dari URL/body harus diverifikasi lewat Policy, bukan diasumsikan valid
Middleware Role Saja Tidak Cukup
Middleware role:admin dari Minggu 2 memastikan "user ini admin", tapi Policy memastikan "user ini pemilik resource spesifik ini" — keduanya dibutuhkan untuk kasus berbeda.

A05: Security Misconfiguration

Konfigurasi yang salah atau lupa diubah dari default saat rilis ke production.

Checklist Konfigurasi Production

.env (production)
APP_ENV=production
APP_DEBUG=false   # WAJIB false — kalau true, stack trace lengkap tampil ke publik
APP_URL=https://laravel-crud-app.com

SESSION_SECURE_COOKIE=true   # cookie hanya dikirim lewat HTTPS
SESSION_HTTP_ONLY=true       # cookie tidak bisa diakses JavaScript (anti-XSS)
Bahaya APP_DEBUG=true di Production
Kalau error terjadi, halaman debug Laravel menampilkan isi .env, query SQL lengkap, dan path server — informasi emas untuk penyerang. Selalu pastikan false di production.

A06: Vulnerable & Outdated Components

Package pihak ketiga (Composer/NPM) yang punya celah keamanan diketahui publik, tapi belum di-update di project.

Cek Kerentanan Dependency

Terminal
# Composer bawaan sejak versi 2.4+
composer audit

# NPM untuk dependency frontend
npm audit

Otomatisasi Cek Berkala

Jalankan composer audit sebagai bagian dari CI/CD (lihat Minggu 8) supaya kerentanan terdeteksi otomatis setiap ada perubahan kode, bukan menunggu diingat manual.

Kebiasaan Sehat
Update composer.lock/package-lock.json secara rutin (bulanan), bukan hanya saat butuh fitur baru. Semakin lama tidak update, semakin besar risiko dan semakin sulit migrasinya.

Enkripsi Data dengan Crypt

Laravel menyediakan facade Crypt untuk enkripsi simetris (bisa dienkripsi dan didekripsi kembali) menggunakan APP_KEY di .env sebagai kuncinya.

Enkripsi Manual

use Illuminate\Support\Facades\Crypt;

$encrypted = Crypt::encryptString('nomor-rekening-sensitif');
$decrypted = Crypt::decryptString($encrypted);

Encrypted Cast di Model (Lebih Praktis)

Untuk kolom yang harus selalu terenkripsi di database, gunakan cast encrypted — Laravel otomatis enkripsi saat simpan, dekripsi saat dibaca.

app/Models/Pelanggan.php
protected function casts(): array
{
    return [
        'nomor_ktp'      => 'encrypted',
        'catatan_khusus' => 'encrypted',
    ];
}
$pelanggan = new Pelanggan();
$pelanggan->nomor_ktp = '3201xxxxxxxxxxxx';   // otomatis terenkripsi saat save()
$pelanggan->save();

echo $pelanggan->nomor_ktp;   // otomatis terdekripsi saat dibaca, transparan
Kolom Terenkripsi Tidak Bisa Di-WHERE
Karena nilai di database benar-benar ciphertext acak, kamu tidak bisa WHERE nomor_ktp = '...' langsung. Kalau butuh pencarian, simpan hash terpisah (misal SHA-256) di kolom tambahan untuk pencarian, sementara data asli tetap di kolom terenkripsi.

Hashing vs Encryption — Jangan Tertukar

AspekHashingEncryption
Bisa dibalik?Tidak (satu arah)Ya (dengan key yang benar)
Dipakai untukPassword, integritas dataData yang perlu dibaca kembali (nomor KTP, catatan medis)
Fungsi LaravelHash::make(), Hash::check()Crypt::encryptString(), cast encrypted
Kalau di-decodeTetap tidak bisa dapat nilai asliBisa didapat nilai aslinya kembali
Password — SELALU Hash, Jangan Pernah Encrypt
// BENAR
$user->password = Hash::make($request->password);

// SALAH — password bisa "dibuka kembali" kalau APP_KEY bocor
$user->password = Crypt::encryptString($request->password);
Kenapa Password Wajib Di-hash, Bukan Di-encrypt?
Kalau APP_KEY bocor, semua data yang di-encrypt bisa didekripsi balik oleh penyerang. Password yang di-hash tidak bisa dikembalikan ke bentuk asli sama sekali, bahkan oleh developer aplikasi sendiri.

Mengelola Secrets

Secrets adalah kredensial sensitif: APP_KEY, password database, API key pihak ketiga, DSN Sentry, dll — semuanya hidup di .env.

Aturan Dasar

  • .env wajib masuk .gitignore — jangan pernah commit ke Git
  • Commit .env.example (tanpa nilai asli) sebagai template untuk tim
  • Generate APP_KEY unik per environment: php artisan key:generate
  • Rotasi secrets secara berkala, terutama setelah ada anggota tim yang keluar

Level Selanjutnya — Secrets Manager

Untuk tim/production yang lebih besar, secrets sebaiknya tidak hanya berupa file .env di server, tapi dikelola lewat layanan khusus.

LayananKelebihan
AWS Secrets ManagerRotasi otomatis, audit log siapa mengakses apa
HashiCorp VaultSelf-hosted, dynamic secrets (kredensial DB sementara)
Laravel Forge / Vapor EnvironmentTerintegrasi langsung dengan deployment Laravel
Untuk Project Skala Kecil-Menengah
.env yang dikelola rapi (tidak commit, akses server dibatasi SSH key) sudah cukup. Secrets manager terpisah baru krusial saat tim besar atau butuh compliance (misal ISO 27001, SOC 2).

HTTPS & Security Headers

Paksa HTTPS

app/Providers/AppServiceProvider.php
use Illuminate\Support\Facades\URL;

public function boot(): void
{
    if ($this->app->environment('production')) {
        URL::forceScheme('https');
    }
}

Middleware Security Headers

Terminal
php artisan make:middleware SecurityHeaders
app/Http/Middleware/SecurityHeaders.php
public function handle($request, Closure $next)
{
    $response = $next($request);

    $response->headers->set('X-Frame-Options', 'DENY');
    $response->headers->set('X-Content-Type-Options', 'nosniff');
    $response->headers->set('Referrer-Policy', 'strict-origin-when-cross-origin');
    $response->headers->set(
        'Strict-Transport-Security',
        'max-age=31536000; includeSubDomains'
    );

    return $response;
}
HeaderMencegah
X-Frame-Options: DENYClickjacking — situs dimuat dalam <iframe> jahat
X-Content-Type-Options: nosniffBrowser "menebak" tipe file, membuka celah XSS
Strict-Transport-Security (HSTS)Browser dipaksa selalu pakai HTTPS, walau user ketik http://
Content-Security-PolicyBatasi sumber script/style yang boleh dimuat halaman

Konfigurasi CORS

CORS (Cross-Origin Resource Sharing) mengatur domain mana saja yang boleh memanggil API kamu langsung dari browser (JavaScript fetch/axios).

config/cors.php
'paths' => ['api/*'],

'allowed_methods' => ['*'],

// Produksi: sebutkan domain spesifik, JANGAN pakai '*'
'allowed_origins' => [
    'https://laravel-crud-app.com',
    'https://admin.laravel-crud-app.com',
],

'allowed_headers' => ['*'],

'supports_credentials' => true,   // perlu true untuk Sanctum SPA auth
Jangan Pakai allowed_origins: ['*'] di Production
Wildcard * berarti semua website di internet boleh memanggil API kamu langsung dari browser user. Kombinasikan dengan supports_credentials: true bahkan bisa membocorkan data user yang sedang login ke situs manapun.

Mencegah SSRF (Server-Side Request Forgery)

SSRF terjadi kalau server dipaksa melakukan request ke alamat internal yang seharusnya tidak bisa diakses publik — misal lewat fitur "import gambar dari URL".

Rentan
public function importGambar(Request $request)
{
    // Penyerang bisa isi url=http://169.254.169.254/latest/meta-data/ (metadata cloud!)
    $isi = file_get_contents($request->url);
    return $isi;
}
Aman — Validasi & Whitelist
public function importGambar(Request $request)
{
    $request->validate(['url' => 'required|url']);

    $host = parse_url($request->url, PHP_URL_HOST);
    $allowedHosts = ['cdn.trustedpartner.com'];

    if (!in_array($host, $allowedHosts, true)) {
        abort(422, 'Domain sumber gambar tidak diizinkan.');
    }

    // lanjut proses download dari domain yang sudah divalidasi
}
Blokir IP Internal Juga
Selain whitelist domain, tolak juga request ke IP privat (127.0.0.1, 169.254.169.254, 10.0.0.0/8, 192.168.0.0/16) supaya fitur seperti "import dari URL" tidak disalahgunakan untuk memindai jaringan internal server.
